Error Identification

3.3.1 Error Identification requires that if an input error is automatically detected, the item that is in error must be identified, and the error must be described to the user in text. This allows users to understand what went wrong and how to fix it, particularly those using assistive technologies.

Briefing and Intent

The intent of this success criterion is to help users understand when a mistake has occurred and how to correct it. If a user submits a form with errors—such as leaving a required field blank or entering invalid data—they must be clearly informed of which field is affected and what needs to be corrected.

Benefits

Clearly identifying errors helps all users, especially those with cognitive disabilities, low vision, or who use screen readers. Providing a text explanation ensures users understand what happened and how to proceed. It also reduces frustration, abandonment, and the likelihood of repeated errors.

Key Requirements

  • Text Descriptions: When an input error is detected, a clear, specific message must be provided in text near the input or at the top of the form.
  • Error Association: The error message must be programmatically associated with the input field (e.g., using aria-describedby or inline labels).
  • Identify the Error: Visually indicate which field contains the error, such as with a visible border and clear focus styling.
  • Accessible Alerts: Ensure that screen reader users are alerted to the error messages using ARIA roles or live regions if applicable.

Examples

  • Required Field: A user submits a form without filling in the required email address. The field is outlined in red and an inline text message reads, “Email address is required.”
  • Invalid Format: The phone number field only accepts digits. An inline message reads, “Phone number requires a 10-digit number with no spaces or dashes.”
  • Programmatic Association: The error message for a password field is linked using aria-describedby so that screen readers announce it when the user returns to the field.
  • Multiple Errors: A form displays a summary at the top listing all detected errors. Each error identifies the field by name and includes the reason for the error—for example: “Phone Number requires a 10-digit number with no spaces or dashes,” and “Email Address is required.” Each item in the summary is an anchor link that moves focus to the corresponding field in the form.

Designing for Error Identification

  • Error Message Placement: Always place error messages adjacent to the input field that is in error. For forms with multiple errors, use a clearly labeled summary at the top of the form that identifies each field by name and describes why it's in error.
  • Contrast Requirements: If red is used for error text or input borders, ensure it meets WCAG 2.1 AA color contrast requirements for regular-sized text (4.5:1). Reference the Text Contrast page in the repository for full guidance.
  • Form States in Design: Designers must show both the static/default state and error state of form components in Figma. This helps ensure developers implement visual error handling accurately and consistently.
  • Format Guidance: Clearly indicate required input formats (e.g., date, phone number) either in:
    • The field label itself
    • Helper or hint text adjacent to the input
    • The error message itself when the format is invalid

By addressing these details in your Figma files, you’ll ensure that error messages are accessible, understandable, and aligned with WCAG 3.3.1 Error Identification requirements.

Testing and Annotating for Error Identification

  1. Review all required input fields and confirm that error states are shown in the Figma design when fields are left blank or contain invalid values.
  2. Check that each error message identifies the field by name and clearly describes the issue (e.g., missing input or invalid format).
  3. Confirm that error messages are placed directly adjacent to the input field in error, except when included in a summary at the top of the form.
  4. Ensure that multiple error summaries include each field name and reason for the error. These should not rely solely on color to convey meaning.
  5. Flag any designs that do not include error styling (e.g., borders, messages) or fail contrast requirements.

Be sure to annotate in the Figma file with specific notes to developers for error message placement, content, and association. Document findings in your walkthrough and confirm all expected behavior is represented visually.

Error Identification

  • Show error message adjacent to the input field in error.
  • Error message must name the field and clearly describe the issue.
  • Associate error message programmatically using aria-describedby or equivalent.
  • Move focus to the first field in error on submission failure.
  • Do not rely on color alone—use visible text and semantic markup.

Developing with Error Identification Considerations

  • Detect Input Errors: Implement client-side and/or server-side validation to detect errors such as missing required fields or incorrect data formats.
  • Describe the Error in Text: When an error is detected, display a clear, descriptive error message that identifies the field by name and explains what is wrong.
  • Programmatic Association: Use aria-describedby or link the <label> and error message to ensure screen readers announce the error when the user focuses on the field.
  • Focus and Visibility: Move focus to the first field in error after submission, and ensure error messages are placed next to the field and styled to be clearly visible.
  • Do Not Rely on Color Alone: Use visible text and semantic HTML to indicate errors. If using color to highlight an error (e.g., red borders), also include icons, messages, or helper text to ensure full accessibility.

Meeting 3.3.1 ensures that all users, including those using screen readers, keyboard navigation, or with cognitive disabilities, can understand what went wrong and how to correct it.
