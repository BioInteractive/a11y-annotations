Keyboard

2.1.1 Keyboard All functionality of the content is operable through a keyboard interface without requiring specific timings for individual keystrokes, except where the underlying function requires input that depends on the path of the user's movement and not just the endpoints.

  • This exception relates to the underlying function, not the input technique. For example, if using handwriting to enter text, the input technique (handwriting) requires path-dependent input but the underlying function (text input) does not.
  • This does not forbid and should not discourage providing mouse input or other input methods in addition to keyboard operation.

2.1.2 No Keyboard Trap This success criterion ensures that if keyboard focus can be moved to a component of the page using a keyboard interface, it can also be moved away from that component using only a keyboard interface. This is crucial for ensuring that users who rely on keyboards to navigate can use the page effectively without getting stuck on any part of it.

  • If keyboard focus can be moved to a component, it must also be possible to move it away using just the keyboard. This helps prevent users from getting trapped in a specific part of the page.
  • If moving focus away requires more than the basic arrow or tab keys, or other standard methods, users must be informed of how to do this. This ensures users are aware of how to navigate freely without encountering barriers.

2.1.4 Character Key Shortcuts This success criterion ensures that if a keyboard shortcut is implemented using a single character key, it can be turned off, remapped, or activated only on focus. This is critical for users who may inadvertently trigger shortcuts or have difficulty using them, such as those with motor disabilities or screen reader users.

  • If single character key shortcuts are available, users should be able to turn them off or remap them to avoid accidental activation.
  • Shortcuts should only activate when the specific interface component has focus to prevent conflicts with assistive technologies or other user inputs.
  • Ensure that users are aware of the existence of these shortcuts and provide a way to customize or disable them as needed.

Designing for Keyboard Accessibility

  • Design for Keyboard and Mouse Users: Ensure that all interactive elements are designed to be accessible for both keyboard and mouse users. Avoid creating features that rely solely on mouse hover interactions.
  • Consider Hover and Focus Interactions: Design for an equivalent experience. If something is triggered on hover it must also be triggered on focus.
  • Accessible Menus and Modals: When designing menus, modals, or other interactive components, ensure that they can be navigated using a keyboard. Include clear instructions or design cues for keyboard navigation.
  • Feedback for Interactive Elements: Provide visual feedback for interactive elements upon focus or activation to indicate that an action is available or has been taken, which is especially important for users navigating with a keyboard.

By considering keyboard users from the initial design phase in Figma, designers can create more inclusive and accessible digital experiences that accommodate a wider range of users.

  1. Visual Inspection: Thoroughly review Figma design annotations to identify interactive elements like buttons, links, and form controls. Ensure these elements are clearly visible and distinguishable in the design, facilitating keyboard navigation.
  2. Keyboard Operability Annotation: Embed detailed annotations or notes within your Figma designs that specify the expected keyboard operability for each interactive element. Highlight the necessity for all elements to be navigable and operable solely via keyboard, emphasizing that specific timings for keystrokes are not required.
  3. Hover and Focus States: Verify that the design specifies hover and focus states for all interactive elements. This is crucial for users navigating via keyboard, ensuring they receive visual feedback similar to mouse users.
  4. Documentation for Developers: Prepare comprehensive notes or documentation outlining keyboard accessibility requirements to accompany Figma design hand-offs to developers. This guidance should include specifics on implementing focusable elements, managing focus order, and ensuring that custom widgets are fully keyboard accessible.
  5. Testing Strategies: Suggest methods or tools for testing keyboard accessibility within Figma or after implementation. This could include manual keyboard navigation tests and the use of automated tools to check focus order and keyboard functionality.

Implementing Keyboard Accessibility

  • Ensure All Interactive Elements Are Keyboard-Operable: Every interactive element that functions with a pointer device must also be operable with a keyboard. This includes links, buttons, form elements, and custom widgets.
  • Equal Functionality: If an element offers functionality to pointer device users, ensure equivalent functionality is available for keyboard users. This may involve adding specific event listeners for key events to interactive elements.
  • No Timing Dependency: All functionalities must be operable via a keyboard interface without relying on specific timings for keystrokes, accommodating users who require more time to type or navigate.
  • Manage Focus for Modals and Dialogs: Implement mechanisms to trap focus within modal dialogs or complex widgets when active, but also provide an easy method for users to escape (untrap focus) using only the keyboard.
  • Avoid Keyboard Traps: Ensure that users can navigate to and from all interactive elements without getting stuck in any part of the content. Provide clear mechanisms for users to move focus away from any trapped area using standard keyboard inputs like Tab or Escape.
  • Implement Character Key Shortcuts Responsibly:
    • If single-character key shortcuts are used, ensure users can turn them off, remap them, or limit their activation to when a specific interface component is focused.
    • Be mindful of conflicts with assistive technologies and other user inputs. Shortcuts should not interfere with screen readers or other keyboard commands.
    • Provide a clear method in settings or controls for users to customize or disable shortcuts as needed.
  • Comprehensive Testing: Test all web content rigorously for keyboard accessibility. This includes manual testing by navigating through the interface using only the keyboard, as well as using automated tools to identify and resolve potential issues.

Adhering to these guidelines ensures your website or application meets WCAG 2.1.1, 2.1.2, and 2.1.4 requirements, making it accessible and user-friendly for those relying on keyboard navigation.
