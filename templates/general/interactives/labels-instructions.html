Labels or Instructions

3.3.2 Labels or Instructions requires that labels or instructions are provided when user input is required. This ensures users understand what information is expected before they interact with a form or input field.

Briefing and Intent

The intent of this success criterion is to help users avoid input errors by providing clear, descriptive labels and guidance before user interaction begins. Instructions should clarify what to enter, how it should be formatted, and whether a field is required.

Benefits

Clear labels and instructions help all users complete forms successfully, particularly users with cognitive, language, or memory disabilities. They also assist screen reader users in understanding form requirements without relying on visual layout alone.

Key Requirements

  • Descriptive Labels: Each input field must have a visible label that clearly describes the expected input (e.g., “Email Address,” “Date of Birth”).
  • Indicating Required Fields: If the form contains a mix of required and optional fields, clearly identify which fields are required. This can be done by:
    • Adding an asterisk (*) and a visual note explaining what it means
    • Using aria-required="true" for programmatic indication
    • Including "(required)" in the label
    If all fields are required, it is acceptable to include a statement at the top such as “All fields are required unless marked optional,” and only mark the optional fields individually.
  • Indicating Optional Fields: If you prefer to highlight optional fields instead, that is also acceptable. Just be sure that users understand the pattern—for example, by appending “(optional)” to those field labels. This method works as long as it’s applied clearly and consistently.
  • Input Format Instructions: When specific formatting is required (e.g., “MM/DD/YYYY”), include that information in:
    • The label itself
    • Helper or hint text next to the field
    • The error message if the input is invalid
  • Proximity and Placement: Place labels, hint text, and format instructions close to the relevant input field. Avoid placing them after other unrelated content.
  • Programmatic Association: Use semantic HTML such as <label for> or aria-labelledby to associate each input with its label and helper text. This is critical for screen reader users.

Examples

  • Email Label: An input field has a visible label reading “Email Address” and is associated with the input via the for attribute.
  • Date Format: A date input field is labeled “Date of Birth” and includes hint text that reads “MM/DD/YYYY.”
  • Required Indicator: A required phone number field includes an asterisk next to the label and the input uses aria-required="true".
  • Grouped Instructions: A group of radio buttons is preceded by the instruction “Select your preferred contact method,” helping users understand the context.

Designing for Labels or Instructions

  • Field Labels: Every input field must include a visible, descriptive label that clearly communicates what information is required.
  • Placement: Place labels above or directly beside the input field to ensure clarity. Avoid floating labels that may disappear once a user starts typing.
  • Required or Optional Indicators: If your form contains a mix of required and optional fields, clearly indicate which fields are required (e.g., asterisk with a legend or label text). Alternatively, you may choose to label optional fields instead—but this must be consistent across the form.
  • Formatting Hints: For fields with specific format requirements (e.g., date, phone number), provide guidance through helper text or hint text shown near the input—before the user makes a mistake.
  • Grouped Instructions: For radio buttons, checkboxes, or grouped inputs, provide a clear group label and any relevant instructions above the group (e.g., “Select your preferred contact method”).
  • Show Default and Error States: Include both the default and error state of form fields in your Figma files so that developers know how each state should look and behave.

Designing with clear, visible labels and instructional text helps prevent errors and supports a wide range of users, including screen reader users and individuals with cognitive disabilities. These practices ensure conformance with WCAG 3.3.2 Labels or Instructions.

Testing and Annotating for Labels or Instructions

  1. Review each input, select, and textarea element to confirm it has a clear, visible label.
  2. Verify that any input field with a specific required format includes visible guidance (e.g., “MM/DD/YYYY”).
  3. If the form has a mix of required and optional fields, check that required fields are indicated consistently (either through asterisks or label text).
  4. If all fields are required, ensure this is clearly stated at the top of the form and that optional fields are labeled accordingly.
  5. Ensure labels are positioned close to their fields (above or beside) and are not visually ambiguous or floating in ways that may disappear or detach.
  6. Check that grouped inputs (radio buttons, checkboxes) include a clear group label and any applicable instructions.
  7. Flag any missing helper text, unclear field names, or insufficient instructions for fields that require a specific input format.

Document findings in your walkthrough and annotate directly in the Figma file with specific guidance for developers. Confirm that all label and instruction behavior is visually represented.

Labels or Instructions

  • Use a visible <label> element for every input field. Do not rely on placeholder text as the only label.
  • Associate helper or hint text with fields using aria-describedby to support screen reader users.
  • Indicate whether fields are required or optional. You can use aria-required="true" and visually show required or optional text. The method must be consistent across the form.
  • Place all instructions and formatting guidance near the field—either before or immediately following the input. Do not use tooltips or hidden content for required format info.
  • Use semantic HTML to group related inputs: <fieldset> with a <legend> for checkboxes or radio button groups. This is required for screen reader users to understand the group label contextually.

Developing with Labels or Instructions Considerations

  • Visible Labels: Use semantic <label for="input-id"> to associate every input with a visible label. Do not rely on placeholders alone.
  • Programmatic Association: Ensure each field is programmatically labeled using label elements, aria-labelledby, or aria-label where necessary.
  • Helper Text: Use aria-describedby to link inputs to helper text or format instructions that appear visually near the field.
  • Required/Optional Indicators: If required fields are not all-encompassing, use aria-required="true" and ensure visual cues (e.g., asterisks or label text) match the form's labeling strategy.
  • Grouped Inputs: Use fieldset and legend to group related checkboxes or radio buttons. Alternatively, associate groups with accessible labels via aria-labelledby.
  • Instruction Placement: Ensure all formatting or input guidance appears before the user makes a mistake—either in helper text or adjacent instructional content.

Following these practices ensures that users—particularly screen reader and keyboard-only users—can understand what’s being asked of them and how to respond, fulfilling WCAG 3.3.2 Labels or Instructions.
