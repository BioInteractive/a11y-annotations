Name, Role, Value

4.1.2 Name, Role, Value ensures that all user interface components have programmatically determined names, roles, and values. These attributes must also be updated appropriately when the component’s state or properties change. This helps assistive technologies convey accurate information to users.

Briefing and Intent

This success criterion ensures that assistive technologies can identify interactive elements on a page and accurately convey their purpose and status to users. Custom components and dynamic widgets require special care to expose the correct name (label), role (function), and value (state or data). Without this, users relying on screen readers or alternative input methods may struggle to understand how to interact with controls or know when something has changed.

Benefits

  • Screen reader users receive accurate labels and context for buttons, inputs, sliders, and dynamic content.
  • Users with cognitive disabilities benefit from consistent labeling and interaction cues.
  • Developers using accessible markup ensure compatibility with a wide range of assistive technologies.

Key Requirements

  • Each interactive component must expose an accessible name, role, and value to assistive technologies.
  • Use semantic HTML whenever possible (e.g., <button>, <input>, <select>).
  • For custom components, use ARIA attributes like aria-label, aria-labelledby, aria-describedby, and role to expose correct metadata.
  • Update accessible attributes dynamically when content or state changes (e.g., a checkbox becomes checked or a form field shows an error).
  • Ensure all roles used are valid and appropriate to the function of the component.

Examples

  • Accessible Toggle: A custom toggle switch has role="switch", aria-checked="true", and aria-label="Enable Notifications", allowing screen readers to announce the correct label and state.
  • Live Status Message: A form updates a confirmation message inside a div with aria-live="polite", so screen readers announce it automatically.
  • Custom Button: A <div> styled as a button includes role="button", tabindex="0", and an aria-label to provide expected behavior and labeling.
  • Slider: A custom range slider includes role="slider", aria-valuemin, aria-valuemax, aria-valuenow, and an accessible label to communicate its state and context.

Definitions

Name
The label or identifier of a user interface component (e.g., button text or aria-label).
Role
The purpose or type of the element (e.g., button, checkbox, slider, dialog).
Value
The current state or data held by the element (e.g., whether a toggle is on/off, the number on a slider, or the text in an input).

Note: Native HTML elements like <button> and <input> meet this requirement by default. This criterion primarily applies to developers building custom interactive components that require additional ARIA markup to expose the correct structure and meaning.

Designing for Name, Role, Value

  • Label All Interactive Elements Clearly: Ensure every interactive element (e.g., buttons, input fields) has a clear, descriptive label. For icons without visible text, include accessible names in the design (e.g., label a search icon as “Search”).
  • Ensure Meaningful Roles: Design interactive components with clear functionality. For example:
    • Buttons should visually indicate they're interactive.
    • Navigation links should be styled distinctly from regular text.
    • Modal dialogs should include clear visual cues for their boundaries and actions.
  • State Management: Provide distinct visual designs for different states (e.g., hover, focus, selected). Ensure that these states are visually obvious and consistent across the interface.
  • Focus Indicators: Include visible focus indicators in all designs for interactive elements. These indicators should meet contrast requirements (e.g., at least 3:1 against the background) and help keyboard users track their position on the page.
  • Dynamic Content Visibility: Ensure elements like tooltips, dropdowns, and modals are visually distinct and follow best practices for when and how they appear.

  1. Visual Identification of Interactive Components: Confirm that all interactive elements (e.g., buttons, form fields, custom widgets) are present in the design and include visible labels where applicable. If any component lacks a label or clear identification, request design updates to ensure they are appropriately labeled.
  2. Annotation for Name Attributes: Use Figma comments to document the accessible name for each interactive element. This includes whether the name will come from visible text, an aria-label, or an aria-labelledby attribute. Ensure names are descriptive and contextually accurate.
  3. Role Verification: Verify that each interactive component's role is visually clear in the design. For example:
    • Buttons should be styled as buttons and function as such.
    • Links should be distinct from buttons and regular text.
    • Custom widgets (e.g., sliders, modals) should have their roles properly defined (e.g., “slider” for a range input, “dialog” for a modal).
    Provide annotations in Figma to clarify expected roles for developers.
  4. Value and State Documentation: Ensure that all interactive elements in the design clearly show their current state or value. For instance:
    • Checkboxes should show checked/unchecked states visually.
    • Sliders should display current values clearly in the design.
    • Any dynamic element, like a dropdown, should indicate expanded or collapsed states.
    Use Figma comments to note how these values or states should be programmatically conveyed to assistive technologies using attributes like aria-checked, aria-expanded, or aria-valuenow.
  5. Documentation for Developers: Provide clear, actionable notes in Figma comments for each interactive element:
    • Specify the accessible name, role, and value for each element.
    • Include instructions on implementing ARIA attributes where necessary.
    • Ensure dynamic elements update their attributes programmatically to reflect state changes accurately.

Name, Role, Value

  • Must provide an accessible name using either visible text or attributes such as aria-label or aria-labelledby.
  • Component must have a clearly defined role appropriate for its function, such as role="button" for buttons, role="checkbox" for checkboxes, or role="navigation" for navigation containers.
  • If the component has a dynamic state or value (e.g., a checkbox being checked/unchecked), the current state must be programmatically conveyed using attributes such as aria-checked, aria-expanded, or aria-valuenow.
  • Ensure all ARIA attributes are updated dynamically when the state of the component changes, so assistive technologies reflect the correct status in real-time.
  • Interactive elements must be focusable and operable using keyboard navigation, ensuring they meet accessibility guidelines for all users.

Implementing Name, Role, Value

To ensure accessible implementation of interactive components, start by assigning a programmatically determinable name to each element. This can be done through visible text or attributes like aria-label or aria-labelledby. For example, a button labeled "Submit" should use either inline text or an aria-label="Submit" to provide an accessible name for assistive technologies.

Next, define the appropriate role for each component. Use semantic HTML whenever possible (e.g., <button>, <input>, <nav>) to ensure elements inherently convey their roles. For custom components, use ARIA roles such as role="button" or role="checkbox" to explicitly define their purpose.

For elements with dynamic states or values (e.g., checkboxes, sliders, modals), ensure their current state is programmatically conveyed. Use attributes like aria-checked for checkboxes, aria-expanded for collapsible elements, or aria-valuenow for sliders. Update these attributes dynamically as the state of the component changes.

Here's an example implementation for a custom toggle button:

Code Example

                            
                              <button id="play-pause-btn" aria-pressed="false" style="width: 60px; height: 60px; padding: 0; font-size: 24px; background-color: #005a9c; color: #ffffff; border: none; border-radius: 50%; cursor: pointer; display: flex; align-items: center; justify-content: center;" aria-label="Play">
                                <span id="icon" aria-hidden="true" style="font-size: 32px;">►</span> 
                              </button>

                              <script>
                                const playPauseBtn = document.getElementById('play-pause-btn');
                                const icon = document.getElementById('icon');
                                playPauseBtn.addEventListener('click', function() {
                                  const isPlaying = playPauseBtn.getAttribute('aria-pressed') === 'true';
                                  playPauseBtn.setAttribute('aria-pressed', !isPlaying);
                                  playPauseBtn.setAttribute('aria-label', isPlaying ? 'Play' : 'Pause');
                                  icon.innerHTML = isPlaying ? '►' : '❚❚'; // Play (▶) and Pause (||)
                                });
                            </script>

Finally, test the implementation using assistive technologies like screen readers to verify that names, roles, and values are correctly communicated. This ensures compliance with WCAG 2.1 AA and provides a seamless experience for users relying on assistive technologies.
