Focus Order

2.4.3 Focus Order requires that when a web page is navigated sequentially, the order of focusable elements must preserve meaning and operability. This ensures that users relying on keyboard navigation or assistive technologies experience content in a logical and predictable sequence.

Briefing and Intent

The intent of this criterion is to make sure that as users tab through the page, the focus moves in a meaningful order that reflects the structure of the content and supports its functionality. An illogical focus sequence can confuse users, cause them to miss critical controls, or create barriers to task completion—especially for users navigating via keyboard, screen reader, or switch device.

Benefits

  • Screen reader and keyboard-only users can move through content in a predictable, meaningful way.
  • Users do not lose track of interactive elements or find themselves trapped in inaccessible areas of the page.
  • Users with cognitive or visual processing disabilities are better able to understand and complete tasks when focus follows expected content structure.

Key Requirements

  • Focusable elements (e.g., links, buttons, form fields) must receive focus in an order that matches the content’s intended reading and interaction flow.
  • Dialogs and modals should receive focus on open, trap focus while active, and return focus to the trigger when closed.
  • Custom interactions (e.g., tabs, accordions) should manage focus so that keyboard users can follow the same flow as mouse users.
  • Programmatic focus shifts should be intentional, not jarring, and should not disorient the user or disrupt expected navigation.
  • Visually reordered elements using CSS (like flexbox or grid) should not break the expected focus order unless meaning is preserved.

Examples

  • Logical Page Layout: A user presses Tab and moves from the site logo to the primary navigation, then to the main call to action—following the visual flow of the page.
  • Modal Behavior: A keyboard user triggers a modal. Focus immediately moves to the modal header, stays within the modal while it’s open, and returns to the trigger button when it closes.
  • Filter Interaction: Activating a “Show filters” button sends focus into the filter panel, and closing the panel returns focus to the button.
  • Dynamic Content: A form dynamically inserts a new input field. That field is placed in the correct position in the DOM so that it becomes the next logical tab stop.

Focus Order Design Considerations

  • Document Focus Flow: In your design documentation, specify the intended focus order to guide development. Highlight how the focus should progress across the page to ensure logical navigation.
  • Visual Indicators for Dialogs: Clearly mark areas in your designs that will become dialogs or modals. Include notes on how focus should be handled, ensuring it remains within these elements while they are active.
  • Collaborate on Focus Strategy: Work closely with the development team to ensure the focus order in your design can be accurately implemented. This collaboration helps maintain the usability and accessibility of the final product.

Providing clear documentation and annotations regarding focus order in your Figma designs will assist developers in creating accessible and operable web content, aligning with WCAG 2.4.3 Focus Order criteria.

  1. Visual Inspection: Examine the Figma designs carefully, paying attention to the layout and order of interactive elements. Ensure that the order in which elements are presented in the design corresponds to a logical and meaningful sequence.
  2. Annotation for Focus Return: Annotate designs where focus should return to a specific element after closing a modal or completing an action, ensuring a seamless user experience.
  3. Tab Order Annotation: Include annotations or notes within your Figma design that describe the expected tab order or focus order. You can use text or comment tools to provide explanations for the intended flow of keyboard focus for more complex designs.
  4. Interactive State Annotations: If your design includes interactive states (e.g., menus, modals, pop-ups), annotate these states to indicate how the focus should behave when these elements are triggered and opened. Describe how the focus should move within and return from these states.
  5. Preparing for Developer Hand-off: Include additional annotations that outline the desired focus order and behavior. This can serve as a reference for developers during implementation.

Focus Order

  • The tab order must follow a logical sequence.
  • Review and adjust the focus order to ensure it follows a left-to-right, top-to-bottom flow that preserves meaning and operability.
  • Ensure that when interacting with links or controls, the next tab stop moves focus to the correct focusable element in a predictable manner.
  • If there are dialogs or pop-ups, confirm that focus is trapped within them when open and returns to the triggering element when closed.

Implementing Focus Order for Accessibility

  • Sequential Focus Navigation: Implement focusable elements in a sequence that reflects the logical flow of the content, ensuring it preserves meaning and operability.
  • Dialog Focus Management: For modals or dialog boxes, ensure focus is trapped within these elements while they're active and returns to the previously focused element upon closure.
  • Focusable Elements Identification: Use proper semantic HTML to mark up focusable components such as links, buttons, and form controls, ensuring they receive focus in the correct order.
  • Custom Focus Handling: When using custom controls or dynamic content, add appropriate ARIA roles and attributes, and use JavaScript to manage focus order and return as needed.
  • Testing and Validation: Regularly test your application with keyboard navigation alone to ensure that the focus order is logical and does not cause confusion or navigation issues.

By carefully managing the focus order and ensuring that all interactive elements are navigable in a logical sequence, developers can greatly enhance the usability and accessibility of web content for keyboard users, aligning with WCAG 2.4.3 Focus Order criteria.
