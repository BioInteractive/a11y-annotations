Error Suggestion

3.3.3 Error Suggestion requires that if an input error is automatically detected and suggestions for correction are known, those suggestions must be provided to the user—unless doing so would compromise the security or purpose of the content.

Briefing and Intent

The intent of this success criterion is to help users successfully complete forms and tasks by providing actionable suggestions when input errors are detected. This improves the user experience by reducing frustration and minimizing trial-and-error attempts—particularly for users with cognitive or language disabilities.

Benefits

Offering helpful suggestions improves form completion rates, reduces abandonment, and supports users who may have trouble with memory, language, or syntax. It is especially valuable for screen reader users and people who may not immediately recognize the cause of an error.

Key Requirements

  • Provide Suggestions: When an error is detected and a fix is known, clearly describe how to correct it in the error message.
  • Field-Specific Guidance: Tailor suggestions to the type of error (e.g., “Date must be in MM/DD/YYYY format” for date fields).
  • Preserve Security: Do not provide suggestions if doing so would expose sensitive information (e.g., password errors).
  • Programmatic Association: Error messages with suggestions should be visually and programmatically associated with the field in error.

Examples

  • Date Format: A user enters “12/31/22” in a date field that requires four-digit years. An error message appears: “Date must be in MM/DD/YYYY format.”
  • Email Validation: A user types “user@email” and the system suggests: “Email must include a domain, such as .com or .org.”
  • Security Exception: A login form does not suggest a correction for an incorrect password to avoid exposing sensitive information. Instead, it displays: “The email or password you entered is incorrect.”
  • Phone Number Guidance: A phone number field returns: “Phone number must include 10 digits—no dashes or spaces.”

Designing for Error Suggestion

  • Actionable Messaging: When an error is likely to occur (e.g., formatting errors), show not just what went wrong, but how to fix it. For example: “Phone number must include 10 digits.”
  • Field-Specific Language: Ensure error messages are written in plain language and reference the specific field, so users are not confused about where the problem occurred.
  • Placement and Association: Place the suggested fix directly below or adjacent to the input field. In multiple-error summaries, include both the field name and suggestion.
  • Show Input Formats: Include formatting hints as helper text before submission (e.g., “Use MM/DD/YYYY”) so users can avoid common errors.
  • Avoid Suggesting for Secure Fields: Do not display suggestions for sensitive fields like passwords. General messages such as “Incorrect email or password” are appropriate.
  • Visual Representation in Figma: Show both the static state and error state of inputs. Include real example text to model the tone and placement of suggestions.

Designing clear, corrective suggestions for form errors improves usability, reduces user frustration, and helps ensure compliance with WCAG 3.3.3 Error Suggestion.

Testing and Annotating for Error Suggestion

  1. Verify that each input field with a known error type includes a corresponding suggested fix in the design mockup.
  2. Confirm error messages use plain language and are specific to the type of error and the field affected.
  3. Ensure messages appear adjacent to the field or in a summary with a field name + suggestion (e.g., “Phone Number must include 10 digits”).
  4. Check that secure fields (e.g., passwords) do not expose correction suggestions that could pose a security risk.
  5. Ensure visual examples of both valid and error states are included in Figma, using realistic message content.

Annotate with specific developer-facing notes to ensure each error message is not only descriptive, but corrective when possible. Link errors programmatically using ARIA where appropriate and verify field naming clarity across the board.

Error Suggestion

  • Provide a suggestion to correct the input if the error type is known.
  • Name the field and describe the correction (e.g., “Phone Number must be 10 digits”).
  • Do not include suggestions for password fields or other secure inputs.
  • Display suggestions visually near the field or in an error summary that includes the field name.
  • Associate the message programmatically using aria-describedby or equivalent method.

Developing with Error Suggestion Considerations

  • Detect and Identify Errors: Use validation logic to identify specific types of input errors (e.g., missing data, incorrect format) and trigger feedback accordingly.
  • Provide Correction Suggestions: If the error type is known and can be corrected, display a suggestion next to the field explaining how to fix it. Examples include formatting instructions or clarifying missing components.
  • Field-Specific Messaging: Tie error messages to the field both visually and programmatically using aria-describedby or aria-live for dynamic feedback.
  • Preserve Security: Avoid offering specific suggestions for sensitive fields such as passwords or secure authentication data. Use neutral messages like “Incorrect email or password.”
  • Consistent Message Placement: Place messages near the field or in a summary, making sure they include the field name and guidance for correction.
  • Use Semantic HTML: Ensure form elements and associated messages are built using semantically correct elements (<label>, <fieldset>, etc.) and styled to be clearly visible.

These practices ensure users receive timely, clear, and actionable feedback when input errors are detected—supporting WCAG 3.3.3 Error Suggestion and improving the form experience for all users.
