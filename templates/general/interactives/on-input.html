On Input

3.2.2 On Input Changing the setting of any user interface component does not automatically cause a change of context unless the user has been advised of the behavior before using the component.

  • No automatic context changes: Adjusting a user interface component should not result in an unexpected change of context, such as navigating to a new page, opening a new window, or submitting a form.
  • Explicit user control: Changes in context must only occur through explicit user actions, like clicking a submit button or confirming a selection.
  • Notification of behavior: If a change in context is necessary, users must be informed of this behavior before they interact with the component.
  • Maintain input stability: Ensure that changes in input do not disrupt the user's workflow or cause unexpected navigation issues.

On Input Design Considerations in Figma

  • Show Different States: Provide examples of different states for user interface components in your Figma designs. Clearly display what happens when a user interacts with an element, ensuring no automatic context changes occur unless explicitly stated.
  • Document User Flows: Clearly document user flows in your design to specify how input changes should be managed. This helps ensure that changes do not disrupt the user's workflow or cause unexpected navigation issues.
  • Explicit User Control: Highlight in your designs that any changes in context, such as navigation or form submissions, require explicit user actions, such as clicking a button, rather than merely adjusting an input.
  • Notification of Behavior: Include notes or annotations indicating where users will be informed about any potential context changes before they interact with components. This ensures users are aware of behaviors that might occur on input.

On Input

When user input triggers changes in content or behavior, ensure the following:

  • Significant changes in context, such as navigation or form submission, should require explicit user actions like clicking a button or pressing Enter. Avoid automatic changes on input.
  • Provide accessible feedback when input triggers changes, using ARIA live regions, alerts, or messages to inform the user of the result.
  • Ensure input changes do not disrupt the user's context or workflow. Avoid navigating away from the current page or opening new windows unless explicitly instructed by the user.
  • Provide clear and accessible instructions and labels for input fields. Use appropriate ARIA attributes to ensure screen readers can announce these instructions.
  • Manage focus appropriately when input changes occur. Direct focus to meaningful elements like validation messages or newly displayed content without disrupting the user's flow.

Implementing On Input Accessibility

  • Explicit User Action: Ensure that any significant changes in context, such as navigation to a new page or opening a new window, require explicit user actions. Avoid automatic context changes upon user input. Use JavaScript event handlers to trigger these actions only when the user explicitly interacts with the element (e.g., clicking a button or pressing Enter).
  • Accessible Feedback: Provide accessible feedback when user input triggers changes. Utilize ARIA live regions to announce updates or changes in content. For example, use aria-live="polite" to announce non-critical updates and aria-live="assertive" for critical updates.
  • Consistent Context: Maintain the user's context when input changes occur. Avoid disrupting the workflow by ensuring that input changes do not automatically navigate away from the current page or open new windows unless explicitly requested by the user.
  • Clear Instructions: Provide clear and accessible instructions for input fields. Use appropriate ARIA attributes such as aria-labelledby and aria-describedby to link labels and instructions to input elements. Ensure screen readers can announce these instructions.
  • Focus Management: Manage focus appropriately when input changes occur. Ensure that focus is directed to meaningful elements, such as validation messages or newly displayed content, without disrupting the user's flow. Use JavaScript to programmatically set focus when necessary.
  • Testing and Validation: Regularly test your application with various input methods (keyboard, mouse, touch) to ensure that input-triggered changes behave as expected. Verify that the focus order remains logical and intuitive, and that users receive appropriate feedback for their input.

By following these guidelines, developers can create accessible and user-friendly web content that aligns with WCAG 3.2.2 On Input criteria.
