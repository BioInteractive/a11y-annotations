Error Prevention

3.3.4 Error Prevention (Legal, Financial, Data) requires that for web pages involving legal commitments, financial transactions, user-controlled data storage, or test submissions, users must be provided mechanisms to review, correct, and confirm before final submission.

Briefing and Intent

The intent of this success criterion is to help users avoid serious consequences that may result from submitting incorrect or unintended information. When interactions involve legally binding agreements, financial data, stored records, or tests, users should have the opportunity to verify their inputs before they are finalized.

Benefits

This ensures a safeguard for users who may make errors due to cognitive limitations, screen reader use, language barriers, or stress. It gives all users confidence that their actions won’t result in irreversible harm or unintended commitments.

Key Requirements

  • Review: Provide a summary or confirmation screen that allows users to review their inputs before final submission.
  • Correct: Give users the option to go back and modify information before committing.
  • Confirm: Include a clear confirmation step (e.g., a “Submit” button after a review screen) to finalize the action.
  • Scope: This applies to pages that trigger legal agreements, financial transactions, save/delete user-generated data, or submit test responses.
  • Exception: This does not apply if the action can be easily reversed (e.g., a temporary shopping cart or draft content).

Examples

  • Bank Transfer: A user reviews recipient details and transfer amount on a confirmation screen before the final submission.
  • Purchase Checkout: An e-commerce checkout includes a full order summary and editable fields before placing the order.
  • Test Submission: An online exam platform prompts users with a “Submit Test” button after showing unanswered questions and responses.
  • Data Deletion: A file deletion interface includes a final “Are you sure?” step with an option to cancel.

Designing for Error Prevention

  • Confirmation Screens: Include a separate review or confirmation screen in your design before final submission for any page that triggers a legal, financial, or data-saving action.
  • Editable Summaries: Design a clear summary of submitted information (e.g., shipping address, payment method) with obvious “Edit” or “Change” links before the final action is confirmed.
  • Reversible Actions: Where applicable, provide undo options or alerts like “You can undo this for the next 30 minutes” for actions such as data deletion or updates.
  • Clear Warnings: Add “Are you sure?” modals or messages when the action is irreversible (e.g., permanent deletion, test submission).
  • Exception Clarity: If the action is easily reversible or doesn’t fall under WCAG 3.3.4 scope, note that visually in your design for audit clarity.
  • State Transitions: In Figma, show the full user flow from form completion → review → confirmation so developers can follow the logic and flow clearly.

By incorporating these elements, your designs will better support users in avoiding irreversible errors and meet the accessibility expectations of WCAG 3.3.4 Error Prevention.

Testing and Annotating for Error Prevention

  1. Review any screens that include legal agreements, financial transactions, user-generated data updates, or test submissions.
  2. Confirm that designs include a review step or confirmation screen before final submission.
  3. Check that users are given an opportunity to correct or change data prior to completing the action.
  4. Look for clear, actionable “Edit,” “Change,” or “Back” links on the review screen or modal.
  5. Verify that irreversible actions include a confirmation message or modal explaining the outcome and offering a way to cancel.
  6. If the action is reversible or doesn’t fall within WCAG 3.3.4 scope, ensure this is noted in the design file.

Annotate with clear guidance on which actions require error prevention patterns. Indicate whether confirmation, correction, or undo opportunities are present. Use language that maps directly to the user's interaction flow.

Error Prevention

  • Include a confirmation screen before submission for legal, financial, or data-impacting actions.
  • Provide a review summary with editable sections or links to update data.
  • Use a clear confirmation step before finalizing the transaction or submission.
  • Add a cancel or undo mechanism for actions like delete or submit where feasible.
  • Clarify when error prevention is not required due to the action being easily reversible.

Developing with Error Prevention Considerations

  • Implement a Review Step: For any action involving legal agreements, financial transactions, data deletion, or test submission, present a summary screen where the user can review all entered data before finalizing the action.
  • Allow Corrections: Provide clear navigation back to previous screens or include "Edit" buttons directly on the review screen to allow changes before submission.
  • Add a Confirmation Step: Ensure a distinct “Confirm” or “Submit” button is used after the review step to prevent accidental submissions.
  • Provide Undo Mechanisms: When appropriate, build in undo options (e.g., “You have 10 minutes to undo this action”) especially for destructive actions like deleting data.
  • Use Accessible Alerts: If using modals or dialogs for confirmations, ensure they follow ARIA modal practices and can be dismissed with keyboard navigation and screen reader support.
  • Document Exceptions: If a submission can be easily reversed and no confirmation step is present, note the reversibility in the codebase and ensure proper messaging is provided to the user.

Following these practices helps users avoid irreversible or harmful actions and aligns your implementation with WCAG 3.3.4 Error Prevention (Legal, Financial, Data).
