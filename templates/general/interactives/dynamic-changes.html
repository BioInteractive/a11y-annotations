Dynamic Chagnes

4.1.2 Name, Role, Value Dynamically updated content needs to have aria-live added to the parent element for the content. This will ensure that screen readers announce the change of content to users.

When updating dynamic content, whether automatically or due to user interaction, it's crucial to ensure that changes are communicated accessibly.

  • Notification Appropriateness: Tailor notifications to the significance of the updates; critical alerts like errors should have higher priority over less critical updates like newsfeed changes.
  • Identification of Updated Content: Mark content that may change as a WAI-ARIA live region using the aria-live attribute, or use special live region roles in certain cases.
  • Awareness of Important Changes: Clearly distinguish between critical updates that require immediate attention and informational updates to avoid disrupting the user unnecessarily, particularly benefiting screen reader users and those with cognitive disabilities.
  • Priority of Update Notifications: Determine the importance of users hearing updates promptly to decide if it should interrupt what the screen reader is currently processing, impacting the value assigned to the aria-live attribute.

Designing for Dynamic Content Accessibility

When designing elements that will dynamically update or insert content, it's essential to visually document both the before and after states in the Figma design. This practice ensures that the transition between states is clear not just for users but also for auditors and developers who will implement these dynamics.

  • Visual Representation of States: Provide clear visual representations in the design files for both the initial (static) state and the post-update (dynamic) state of content. This helps in understanding the visual and functional transition due to dynamic updates.
  • Annotation of Dynamic Changes: Annotate where and how content changes occur within the design. Highlighting these changes aids developers and auditors in identifying functionality that needs special attention during development and testing.
  • Communication of Interactive Dynamics: Clearly mark elements in the Figma design that will undergo dynamic changes. Use annotations to describe what triggers these changes and what the expected outcome is.
  • Demonstration of Accessibility Features: Ensure that all dynamically changing content adheres to accessibility standards by designing for keyboard and screen reader users from the start. Include notes on any specific ARIA roles or properties that will be needed.

By documenting these aspects, designers ensure that dynamic content is both functional and accessible, facilitating a smoother handoff to developers and alignment with accessibility auditors.

  1. Verification of States: Ensure the Figma design illustrates both the static and dynamic states of content. Auditors should check that each state is visually represented to understand how content changes dynamically.
  2. Annotation of Dynamic Behaviors: Review and annotate how dynamic changes are managed within the design. Specify where aria-live attributes should be applied and the preferred politeness level (e.g., "assertive" or "polite") to ensure important updates are communicated appropriately.
  3. Focus Management: Examine the design for focus order and management, especially after dynamic changes. Annotations should clearly indicate the intended focus path, particularly if new content becomes focusable. The notes should suggest placing focus on newly revealed content if it's critical for continuity.
  4. Instructions for Developers: Create detailed instructions within the Figma annotations for developers, outlining how to implement dynamic content updates accessibly. This includes using appropriate ARIA roles and ensuring that dynamic content insertion does not disrupt the user's workflow unless intended.
  5. Testing Dynamic Content: Recommend strategies and tools for testing the accessibility of dynamic content both within Figma and in the implemented environment. Encourage the use of screen readers and accessibility testing tools to simulate user interaction with dynamic updates.

Dynamic Changes

User must be made aware of dynmic changes. There are three main ways to ensure dynamic chagnes are announced to assistive technologies:

Option One: Add a live region

  • Live Region Setup: Add a live region to a container element that will receive dynamic updates.
  • Implementing ARIA Live: Use the aria-live attribute set to "polite" on the container to ensure that changes are announced by assistive technologies at the next available opportunity, without interrupting the user.
  • Dynamic Content Announcements: This setup will allow screen readers to automatically announce updates made within the container, such as changes in content or state triggered by user interactions.

Option Two: Add a visually hidden live region

  • Setup Hidden Live Region: Insert a visually hidden <div> element on the page using a class like "sr-only" to ensure it is not visible to sighted users but accessible to screen readers.
  • Apply ARIA Live: Add aria-live="polite" to the <div> with the "sr-only" class, allowing assistive technologies to announce updates without interrupting the user.
  • Update Announcements: Create an event listener that updates the live region with relevant text, such as "Content updated", whenever dynamic changes occur.
  • Cleanup Text: Use setTimeout to clear the text from the live region after it has been announced, ensuring the region is ready for the next update.

Option Three: Move focus to the newly revealed content

Must use this one if content insert before triggering element

  • Focus Management: When dynamic content is revealed, temporarily shift focus to this new content by adding tabindex="-1" to the element or element's container, ensuring that assistive technologies can properly announce the updates.
  • Screen Reader Announcements: This focus shift ensures that the content of the newly revealed area is announced by screen readers, aiding in better navigation and understanding of the content changes.
  • User Experience Caution: Be cautious when manipulating focus as it can affect the overall user experience and navigation. Always consider the broader impact on accessibility and user interaction.
