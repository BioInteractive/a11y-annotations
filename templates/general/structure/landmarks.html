Landmarks

1.3.1 Info and Relationships Information, structure, and relationships conveyed through presentation can be programmatically determined or are available in text.

Landmarks identify regions of a screen and help people who use assistive technology to make a mental model of a screen's content.

Landmark regions must be present and must be used appropriately to accurately reflect the purpose of the content.

Landmarks

  • banner: A region that contains the prime heading or internal title of a page. Use a <header> or add role="banner".
  • navigation: A region that contains navigation links to other pages or different parts of the same page. Use a <nav> or add role="nav".
  • main: A region that contains a page's main content. Use a <main> or add role="main".
  • region: A region that contains a perceivable section of the page containing content that is sufficiently important for users to be able to navigate to the section. A region landmark isn't exposed as a landmark region unless it has an accessible name. Use a <section> or add role="region".
  • form: A region of the document that represents a collection of form-associated elements, some of which can represent editable values that can be submitted to a server for processing. Use a <form> or add role="form".
  • search: A region of the page containing search functionality. Use a <search> or add role="search".
  • complementary: Any section of the document that supports the main content, yet is separate and meaningful on its own. Use a <aside> or <section> and add role="complementary".
  • contentinfo: A region that contains information about the parent document such as copyrights and links to privacy statements. Use a <footer> or add role="contentinfo".

  1. Install the Plugin:

    Ensure you have the "Include—Accessibility Annotations" plugin installed in your Figma account. You can find and install the plugin from the Figma Community.

  2. Activate the Plugin on Your Figma Board:

    Open the Figma Board that you wish to annotate. Activate the "Include—Accessibility Annotations" plugin for the current board.

  3. Choose Web or Mobile Option:

    Depending on whether you are designing for the web or mobile, choose the appropriate option within the plugin.

  4. Select a Top-Level Frame:

    Ensure that you have selected a top-level frame that represents the main content of your web page or mobile app. This frame should contain the content you want to annotate.

  5. Run the Accessibility Checker:

    Execute the accessibility checker within the plugin. This will scan your selected frame for accessibility issues and provide recommendations.

  6. Add Landmark Outlines:

    After running the accessibility checker, you can add landmark outlines to your Figma Board based on the recommendations from the plugin. Follow the guidelines provided by the plugin to add landmarks like "banner," "navigation," "main," etc., as appropriate.

  7. Provide Short Labels:

    When creating multiple landmarks of the same type (e.g., multiple "region" landmarks), you'll be prompted to provide short labels. Ensure that these short labels are concise and meaningful, but do not include words like 'landmark,' 'region,' or 'navigation' since screen readers already announce the type.

  8. Limitation for Main, Banner, and Contentinfo Landmarks:

    Note that there is a limit of one "main," one "banner," and one "contentinfo" landmark per board. Ensure you follow this limitation while annotating.

  9. Save Your Progress:

    After adding all the required landmarks and labels, save your progress within the plugin. This ensures that your accessibility annotations are retained and can be reviewed and updated as needed.

Accessibility Considerations for Implementing Landmarks

  • Use Semantic HTML5 Elements: Implement semantic elements like <header>, <nav>, <main>, <footer>, and <aside> as landmarks to enhance document structure. These elements are inherently recognized by assistive technologies.
  • ARIA Landmark Roles: Apply ARIA landmark roles (e.g., role="complementary" for side content not directly related to the page's main content) when semantic HTML5 cannot be used, ensuring roles are programmatically determinable.
  • Unique Naming for Landmarks: Use aria-labelledby or aria-label to provide unique names for landmarks when there are multiple instances of the same landmark type, aiding users in navigation.
  • Screen Reader Navigation: Understand that screen reader users rely on shortcut keys to quickly navigate through landmarks (e.g., jumping directly to the main content or navigation), emphasizing the need for proper landmark implementation.
  • Content Within Landmarks: Ensure all content is contained within a landmark role. This practice helps users understand the structure and navigate the content efficiently, avoiding any parts of the page being inaccessible via keyboard shortcuts.

By adhering to these key considerations, developers can create web content that is not only compliant with accessibility standards but also provides a better navigational experience for users relying on assistive technologies.
