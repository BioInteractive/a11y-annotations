1.3.2 Meaningful Sequence (WCAG 2.0 A) When the sequence in which content is presented affects its meaning, a correct reading sequence can be programmatically determined.

1.3.4 Orientation (WCAG 2.1 AA) Content does not restrict its view and operation to a single display orientation, such as portrait or landscape, unless a specific display orientation is essential.

2.4.1 Bypass Blocks (WCAG 2.0 A) A mechanism is available to bypass blocks of content that are repeated on multiple Web pages.

2.4.2 Page Titled (WCAG 2.0 A) Web pages have titles that describe topic or purpose.

2.4.5 Multiple Ways (WCAG 2.0 AA) More than one way is available to locate a Web page within a set of Web pages except where the Web Page is the result of, or a step in, a process.

3.1.1 Language of Page (WCAG 2.0 A) The default human language of each Web page can be programmatically determined.

3.1.2 Language of Parts (WCAG 2.0 AA) The human language of each passage or phrase in the content can be programmatically determined except for proper names, technical terms, words of indeterminate language, and words or phrases that have become part of the vernacular of the immediately surrounding text.

Understanding WCAG 2.0 A Success Criterion 1.3.2: Meaningful Sequence

The WCAG 2.0 A Success Criterion 1.3.2 focuses on ensuring that the content is presented in a way that preserves its meaning, regardless of the presentation order. This is crucial for users who rely on assistive technologies, like screen readers, as these technologies read the content in the order it appears in the HTML document. Ensuring a meaningful sequence aids in comprehending and navigating the content effectively.

Developer Guidelines for Meeting Criterion 1.3.2

  • Use semantic HTML elements to structure the content logically (e.g., <header>, <nav>, <main>, <footer>).
  • Ensure the reading and navigation order is logical and intuitive, especially when CSS is used for layout adjustments.
  • Avoid using CSS to visually reorder elements that disrupts the natural reading sequence.
  • Test with CSS disabled to verify if the content still maintains a logical order.
  • Use ARIA landmarks to define regions of the page if necessary.
  • Test your webpage with screen readers to ensure the content is read in a meaningful order.

Understanding WCAG 2.1 AA Success Criterion 1.3.4: Orientation

The WCAG 2.1 AA Success Criterion 1.3.4 addresses the need for content to be accessible in both portrait and landscape orientations. This criterion is essential for ensuring that content can be used by everyone, regardless of how their device is used or mounted. It's particularly important for people who use devices in a fixed orientation, such as a tablet attached to a wheelchair.

Developer Guidelines for Meeting Criterion 1.3.4

  • Design and develop web content that works in both portrait and landscape orientations.
  • Avoid locking the orientation to either portrait or landscape unless necessary for specific functionalities.
  • Ensure that all functionality and content are available in both orientations.
  • Use responsive design techniques to adapt the layout to different screen orientations.
  • Test your website or application on various devices to ensure proper functioning in both orientations.
  • Consider the use of CSS media queries to adjust styles and layouts based on the orientation.

WCAG 2.0 A Success Criterion 2.4.1: Bypass Blocks

Success Criterion 2.4.1 Bypass Blocks, a key aspect of the WCAG 2.0 A guidelines, focuses on providing mechanisms for users to bypass repetitive content. This is especially important for users who rely on screen readers or keyboard navigation. Common repetitive content includes navigation menus, lengthy lists of links, and recurring site-wide components like carousels or sidebars.

Appropriate Use of Skip Links and Identifying Bypass Blocks

  • Skip Links: Implement skip links at the beginning of the page to allow users to jump directly to the main content or other key sections. Skip links are particularly useful for bypassing repetitive navigation menus and headers.
  • Visibility of Skip Links: Ensure that skip links are visible when focused and easily accessible via keyboard navigation. This enhances usability for those who rely on keyboard control.
  • ARIA Landmarks: Use ARIA landmarks to mark regions like navigation, main, aside (for sidebars), and footer. This helps in providing a roadmap of the page for screen reader users.
  • Logical HTML Structure: Organize your HTML structure logically. Where feasible, place the main content first in the HTML source, especially if the navigation menu is extensive.
  • Identifying Common Bypass Blocks: Recognize common areas where skip links are beneficial. This includes:
    • Repetitive navigational elements like top bars or sidebar menus.
    • Recurring elements like carousels, especially when they contain multiple items or links.
    • Long lists of links or lengthy introductory content before the main information.
  • Testing for Effectiveness: Regularly test your website with screen readers and keyboard navigation to ensure the skip links and other bypass mechanisms are functioning correctly and are effectively aiding navigation.

How to Develop Skip Links

  1. Step 1: Identify the Correct Target
    1. Determine the Main Content Area: Identify the primary content section for navigation. Wrap it in a semantic HTML element like <main> or use a meaningful ID.
    2. Use Semantic HTML: Ensure that this area is clearly marked and accessible.
  2. Step 2: Update Skip Link's Href Attribute
    1. Adjust the href Attribute: The href of the skip link should point to the ID of the main content container, like href="#main-content".
    2. Ensure Unique IDs: Confirm that the ID is unique within the page.
  3. Step 3: Verify Visual and Functional Clarity
    1. Test Visibility: Make the skip link visible upon activation, usually at the top of the page.
    2. Check Focus: Ensure the focus shifts to the main content or its first interactive element.
  4. Step 4: Implement Consistency Across Pages
    1. Standardize Skip Links: Maintain consistency in skip link phrasing, placement, and targets across all pages.
    2. Template Updates: For sites using templates, include the skip link and its target in the template to ensure uniformity across pages.

WCAG 2.0 A Success Criterion 2.4.2: Page Titled

Success Criterion 2.4.2 Page Titled, under WCAG 2.0 A guidelines, emphasizes the importance of descriptive and informative page titles. A well-defined page title helps users, particularly those with assistive technologies, to understand the content and context of the webpage. It is a crucial aspect of web accessibility, offering initial orientation and aiding efficient navigation.

Development Guidelines for Criterion 2.4.2

  • Ensure each webpage has a descriptive <title> element that accurately describes the topic or purpose of the page.
  • Place the <title> element within the page's <head> section.
  • Keep the title concise yet descriptive enough to provide information about the page content at a glance.
  • For websites with multiple pages, start the title with unique information, followed by more general information, like the site name.
  • Regularly review and update titles to ensure they remain relevant and accurate as content changes.
  • Avoid using vague titles like "Untitled" or "New Page," which offer no meaningful information about the page content.

Additionally, page titles can be effectively used to reflect dynamic changes on the webpage, such as error or success messages following form submissions. For instance, updating the page title to indicate a submission error or success can immediately alert users with screen readers about the status of their action. This practice enhances the user experience by providing immediate, clear feedback directly in the page title, especially in complex forms or multi-step processes.

WCAG 2.0 AA Success Criterion 2.4.5: Multiple Ways

Success Criterion 2.4.5 Multiple Ways, under WCAG 2.0 AA, requires that there are multiple ways for users to find web pages within a set of web pages, except where the webpage is the result of, or a step in, a process. This criterion is often met by providing primary navigation, along with additional ways like a site map or a search component. This ensures users have various methods to navigate and locate content, enhancing accessibility for a diverse range of users.

Development Guidelines for Criterion 2.4.5

  • Include a consistent, primary navigation menu across all webpages.
  • Provide a site map that lists and links to all accessible pages, especially for large or complex websites.
  • Implement a search functionality to help users find content more directly.
  • Consider adding additional navigation aids, such as breadcrumb trails or category lists.
  • Ensure that alternative navigation methods are easy to find and use.
  • Regularly update and maintain alternative navigation methods to ensure accuracy and relevance.

WCAG 2.0 A Success Criterion 3.1.1: Language of Page

Success Criterion 3.1.1 Language of Page is a part of the WCAG 2.0 A guidelines. It mandates that the primary language of each webpage should be programmatically determined. This is crucial for users who utilize text-to-speech software or screen readers, as it ensures that text is read and pronounced in the correct language. Properly specifying the language of the page aids in comprehension and accessibility for a global audience.

Development Guidelines for Criterion 3.1.1

  • Use the lang attribute in the <html> tag to specify the primary language of the webpage.
  • Choose the appropriate language code for the lang attribute based on the primary content language of the page.
  • For pages containing multiple languages, identify the primary language of the page in the lang attribute and use additional lang attributes for content sections in different languages.
  • Ensure that metadata, such as keywords and descriptions, also align with the specified page language.
  • Regularly review the language specification to ensure it remains accurate with content updates.

WCAG 2.0 AA Success Criterion 3.1.2: Language of Parts

Success Criterion 3.1.2 Language of Parts, as outlined in WCAG 2.0 AA, requires that the language of each passage or phrase in the content can be programmatically determined. This is especially important for content that includes multiple languages. Properly indicating the language for each section helps users with assistive technologies, such as screen readers, to correctly interpret and pronounce the text.

Development Guidelines for Criterion 3.1.2

  • Use the lang attribute on elements containing text in a language different from the main page language.
  • Identify and mark every section or phrase with a different language accurately.
  • Ensure that changes in language are clearly identified in the code, so assistive technologies can adjust pronunciation and reading accordingly.
  • Regularly check the content to ensure that all parts in different languages are correctly marked, especially after updates or additions.
