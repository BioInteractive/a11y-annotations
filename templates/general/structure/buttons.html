Buttons

Button Vs. Link

A button <button> performs an action. (Open Disclosure, Play Video, Launch Dialog)

A link <a> navigates. (In-page anchor link, internal/external URL)

1.3.1 Info and Relationships: This criterion ensures that information, structure, and relationships conveyed through presentation can be programmatically determined or are available in text, crucial for users of assistive technologies. Specifically for buttons, it highlights the need for:

  • Semantic use of the <button> element to convey the role automatically to assistive technologies.
  • Accessible labels that clearly describe the purpose of the button, especially for icon buttons or those without visible text.
  • Distinct styling to indicate their functionality as interactive elements separate from other content like links or text.

Ensuring buttons are perceivable as buttons, have an accessible name, are included in focus order, and function consistently for all users are key to complying with this criterion.

4.1.2 Name, Role, Value: This criterion focuses on ensuring that for all user interface components, including buttons, the name and role can be programmatically determined; states, properties, and values that can be set by the user can be programmatically set; and changes are announced by assistive technologies. For buttons, this includes:

  • Ensuring the role of a button is understood, especially when using custom elements that act as buttons.
  • Providing a name through visible text or accessible attributes like aria-label or aria-labelledby.
  • Exposing any state changes (e.g., pressed, expanded) to assistive technologies using appropriate ARIA attributes.

Accessibility Considerations for Button Design in Figma

Ensuring buttons meet WCAG criteria 1.3.1 (Info and Relationships) and 4.1.2 (Name, Role, Value) includes considerations for both text and icon-based buttons:

  • Distinguishable Buttons: Design buttons to be clearly distinguishable from links or non-interactive elements. Use shape, elevation, or texture in addition to color.
  • Icon Buttons: When a button uses an icon instead of text, ensure it has an accompanying text label accessible to screen readers. This can be achieved through the use of tooltips, aria-labels, or off-screen text noted in the design phase to convey the button's purpose.
  • Descriptive Names for Action: Whether using text or icons, ensure the purpose of the button is clear and descriptive. Avoid ambiguity to make the interface intuitive for all users, including those using assistive technologies.
  • Contrast and Visibility: Ensure sufficient contrast between button icons (or text) and their background, adhering to a minimum contrast ratio of 4.5:1 for text/icons to background. For larger text/icons, a minimum ratio of 3:1 is required.

By incorporating these design considerations into Figma prototypes, designers can create more accessible and user-friendly interfaces that comply with key accessibility standards.

Conducting Accesibility Audit and Annotations

  1. Use this A11y Annotation Kit.
  2. Identify all UI elements that perform actions.
  3. During phase one, ask questions if neccesary to understand functionality.
  4. During phase two in preparation for developer hand-off, add a sticky note annotation to define each button element as a button.
  5. Most buttons will require further annotating for the developer hand-off phase. Please see interactives and the How to Annotate for more detailed instructions on how to number the elements and write annotations specific for more complex components.
  6. If the button does not have a visible text label it is recommended to use the button annotation tag with the additional label to ensure the button name will be communicated to the developer for programming.

Accessibility Considerations for Implementing Buttons in HTML

For developers translating Figma designs into accessible HTML, adhering to WCAG criteria 1.3.1 (Info and Relationships) and 4.1.2 (Name, Role, Value) is crucial, especially for buttons, including those with icons:

  • Semantic HTML: Utilize the <button> element for buttons to ensure they are accessible and their role is understood by assistive technologies. Avoid using <div> or <span> for interactive elements unless necessary.
  • Accessible Name for Icon Buttons: When a button is represented by an icon without visible text, use aria-label or aria-labelledby to provide an accessible name that describes the button's action, ensuring it is comprehensible to screen reader users.
  • Interactive States: Define clear visual and programmable states (hover, active, focus, disabled) for buttons using CSS and ARIA attributes to communicate the state changes to all users, including those using assistive technologies.
  • Contrast and Visibility: Ensure that buttons meet the minimum contrast ratio requirements against their background. Use CSS to achieve a contrast ratio of at least 4.5:1 for text and icons to background. Larger elements require a ratio of at least 3:1.
  • Focus Management: Implement proper focus management to ensure keyboard users can navigate through interactive elements seamlessly. Use tabindex where necessary and ensure custom buttons made from non-semantic elements are fully keyboard accessible.
  • Use of ARIA: Apply ARIA roles and properties judiciously to enhance accessibility when native HTML does not suffice. For instance, aria-pressed for toggle buttons to indicate their state.

By focusing on these development practices, developers can ensure that the buttons in the web interface are accessible, providing a better user experience for everyone, including those relying on assistive technologies.
