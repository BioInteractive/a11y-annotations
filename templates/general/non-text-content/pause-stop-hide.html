Pause, Stop, Hide

2.2.2 Pause, Stop, Hide requires that users be able to pause, stop, or hide any moving, blinking, scrolling, or auto-updating content that starts automatically, lasts more than five seconds, and is presented alongside other content—unless it is essential to the activity.

Briefing and Intent

The goal of this success criterion is to reduce distraction and improve usability for people with cognitive disabilities, vestibular disorders, or attention challenges. Uncontrolled motion can interrupt comprehension, impair concentration, and even cause physical discomfort. This criterion ensures that users maintain control over dynamic content that competes for their attention.

Benefits

Providing control over motion ensures a more inclusive experience for users who may be negatively impacted by visual distractions or animation. It helps users focus on primary content and allows screen reader users to control timing when navigating or reviewing updates in real time.

Key Requirements

  • Moving, Blinking, Scrolling: If it starts automatically, lasts more than five seconds, and is displayed in parallel with other content, users must be able to pause, stop, or hide it—unless it is essential to the activity (e.g., a real-time simulation).
  • Auto-updating: If content updates automatically and appears alongside other content, users must be able to pause, stop, hide it, or control the update frequency, unless the update is essential (e.g., a real-time stock ticker).

Examples

  • Autoplaying Videos: A video that begins playing when the page loads includes a pause button for users to stop it.
  • Carousel or Slider: An image carousel automatically cycles every 5 seconds, but includes controls to pause or navigate slides manually.
  • GIF Animations: An animated GIF is set to only loop once or is accompanied by a control to stop playback.
  • Background Animations: Subtle motion effects in the page background can be toggled off by the user or are disabled entirely if the system has motion reduction settings enabled.

Additional Notes

User-activated animations, such as scrolling effects that only trigger when users interact with the page (e.g., clicking, scrolling, hovering), are not covered under 2.2.2. These do not require pause/stop/hide controls because the user initiates them. However, it’s still important to ensure that such animations are not disruptive.

To enhance accessibility for people with motion sensitivity, developers are strongly encouraged to implement the prefers-reduced-motion CSS media query. This respects user operating system settings and allows animations to be reduced or removed for those who need it.

Designing for Pause, Stop, Hide Accessibility

  • Identify Auto-updating or Moving Elements: Coordinate with content leads to determine if any components (like carousels) will auto-update, move, or blink.
  • Implement Control Mechanisms: Design pause, stop, or hide buttons for auto-updating or moving content to allow users to control these elements.
  • Accessibility Annotations: Annotate designs to highlight these control mechanisms and ensure they are clearly communicated to developers for implementation.
  • Ensure Visibility: Design these controls to be visible and accessible to all users, including those who rely on assistive technologies.

By incorporating these design considerations, you'll contribute to creating accessible content that meets WCAG 2.2.2 Pause, Stop, Hide criteria, enhancing user control and accessibility.

Auditor Checklist for Pause, Stop, Hide Criteria

  1. Identify any auto-updating, moving, or blinking elements in designs. Ask if uncertain about content behavior.
  2. Confirm the presence of mechanisms (pause, stop, hide buttons) for controlling auto-updating or moving content.
  3. Ensure controls for these elements are accessible, easily identifiable, and clearly documented in design annotations.
  4. Flag any elements lacking control mechanisms, especially in cases of auto-scrolling or auto-playing content.
  5. Consult with designers and content leads on implementing accessible control features for dynamic content.

Following these steps helps ensure designs meet WCAG 2.2.2 Pause, Stop, Hide criteria, enhancing accessibility and user control over dynamic content.

Pause, Stop, Hide

  • If there is moving blinking scrolling auto updating or autoplaying content there must be an accessible mechanism provided to pause stop or hide it.
  • Warning: Content that is moving, blinking, scrolling or auto updating does not have a mechanism to pause stop or hide it.
  • Recommendation: Implement a pause play button or other mechanism.

Developer Guidelines for Implementing Pause, Stop, Hide Features

  • Integrate Control Mechanisms: For any auto-updating or moving content, implement accessible controls (e.g., pause, stop, hide buttons) allowing users to manage content display.
  • Ensure Accessibility: Controls should be keyboard navigable and have clear labels for assistive technologies.
  • Test Functionality: Verify that control mechanisms work as expected across different devices and browsers.
  • Documentation: Annotate code to document how and where control mechanisms are implemented for future reference and maintenance.
  • Consult Design Annotations: Work closely with design annotations to accurately implement the envisioned pause, stop, or hide functionalities.

Adhering to these guidelines ensures your web content meets WCAG 2.2.2 Pause, Stop, Hide criteria, enhancing user control and accessibility.
