Accordion

An accordion is a user interface component that allows users to expand and collapse sections of related content. Commonly used for FAQs, filters, settings panels, and progressive disclosure patterns, accordions help organize dense content into manageable segments. They improve scannability and reduce visual clutter while allowing users to control how much information is visible at once. For accessibility, accordions must be built to ensure that all users—including those using keyboards and screen readers—can navigate, understand, and operate each section consistently.

Below is a list of WCAG success criteria requirements and accessibility considerations:

  • Proper Use of ARIA Roles and Properties: Implement role="button" for each accordion header, aria-expanded to indicate the open or closed state of the accordion panel, and aria-controls to link the header to its corresponding panel. ARIA: Accordion Example
  • Keyboard Navigation: Ensure accordion headers are navigable and operable with keyboard alone, including the ability to expand/collapse with Enter/Space keys and navigate between headers with arrow keys. 2.1.1 Keyboard
  • Focus Management: Manage focus within the accordion to ensure a logical tab order and that moving through the component does not trap keyboard users. 2.4.3 Focus Order
  • Panel Role and Labeling: Use role="region" for accordion panels and aria-labelledby to associate each panel with its header, enhancing screen reader user experience. ARIA: region role
  • State Indication: Visually indicate the expanded or collapsed state of each accordion section, in addition to using aria-expanded, to assist users in understanding the accordion's current state. 1.3.1 Info and Relationships
  • Screen Reader Hidden Icons: For icons that indicate the expanded or collapsed state, ensure they are marked with aria-hidden="true" if they are purely decorative or redundant when text labels are provided. 4.1.2 Name, Role, Value
  • Visual Contrast: Design accordion headers with sufficient color contrast between text/icons and their background, ensuring visibility for users with visual impairments. 1.4.3 Contrast (Minimum)

Meaning Behind the Requirement

Accordions help reduce visual clutter, but without clear visual cues, users may miss important content or not realize sections can expand. This impacts users with cognitive, vision, or motor disabilities. Accessible design makes these interactive patterns understandable at a glance.

What to Design

  • Interactive Headers: Design headers to look clickable using consistent button styling like borders, background, or shape.
  • State Indicators: Include visual elements that clearly show whether a section is expanded or collapsed, such as rotating chevrons or other visual changes.
  • Header-Panel Relationship: Keep each panel visually grouped with its header using spacing and alignment to reinforce the connection.

Best Practices

  • Color Contrast: Ensure all text and interactive elements have at least a 4.5:1 contrast ratio against their background.
  • Focus Styling: Include a visible focus ring or highlight for keyboard users navigating to accordion headers.
  • Redundancy: Don't rely on icon-only indicators. Support state changes with layout or visual motion cues as well.
  • Consistent Placement: Place visual indicators in the same spot on every accordion item to avoid confusion.

When auditing an Accordion component please review the content leads documentation and the designer's implementation for alignment and consistency. Then review for accessibility considerations to flag for the designer and then annotate the component requirements for developer handoff.

When auditing the accordion component within a Figma design please consider the following accessibility issues to flag for the designer to review:

  • Visual Clarity of Buttons: Ensure buttons are clearly designed as interactive elements and are easily distinguishable from other content.
  • State Indication Icons: Verify if there are icons indicating the expanded or collapsed states of the accordion, and ensure these icons are visually intuitive.
  • Association Between Button and Panel: Check that each button is clearly associated with its corresponding panel, with visual cues to indicate this relationship.
  • Consistency of Design: Assess the consistency in the design of all accordion components throughout the project to ensure a uniform user experience.
  • Heading Structure: Review if the accordion's design suggests a proper heading structure for the content within each collapsible panel.
  • Legibility: Confirm that text within buttons and panels is legible, with adequate font size and color contrast.
  • Text ContrastButton text must maintain a 4.5:1 minimum contrast with the page background in all states.
  • Icon ContrastThe icon is meaningful visually and must maintain a minimum 3:1 contrast ratio in all states

Accordion

Accordion Buttons

  • Maintain heading structure - if headings have been used
  • Nest <button> in heading tags - if headings have been used
  • Must have toggling aria-expanded="true/false" depending on expanded/collapsed states.
  • Should have aria-controls attribute equal to the ID of the container of the element it controls.
  • Icon should be hidden for screen reader users with aria-hidden="true"

Accordion Panels

  • Must have role="region"
  • An aria-labelledby attribute equal to the ID of the associated button.

As a developer, your goal is to apply ARIA roles, properties, and states accurately—especially those that must update dynamically during interaction. This section includes implementation notes, attribute guidance, and a Bootstrap-ready code example.

ARIA Attributes: Manual vs. Programmatic

  • Manual Attributes (Static in HTML):
    • role="button" on the header <button>
    • aria-controls linking header to panel ID
    • role="region" on panel content (if landmark)
    • aria-labelledby on panel referencing header ID
  • Programmatic Attributes (Updated with JS):
    • aria-expanded="true/false" on header button, toggled based on panel state
    • hidden or class-based visibility toggles for the panel

Accessible Accordion Code Example

This example shows the essential structure and attributes for a single Bootstrap accordion item. JavaScript handles dynamic toggling.

  
  <!-- Accordion Item -->
  <div class="accordion-item">
    <h2 class="accordion-header" id="accordionHeader1">
      <button class="accordion-button collapsed" type="button"
              data-bs-toggle="collapse"
              data-bs-target="#accordionPanel1"
              aria-expanded="false"
              aria-controls="accordionPanel1">
        Accordion Title
      </button>
    </h2>

    <div id="accordionPanel1" class="accordion-collapse collapse"
         role="region"
         aria-labelledby="accordionHeader1"
         data-bs-parent="#accordionGroup">
      <div class="accordion-body">
        Content of the accordion panel.
      </div>
    </div>
  </div>

Tip: Ensure all id values are unique across the page.

Key Developer Considerations

  • State Management: Use JavaScript to toggle aria-expanded and update visibility if not using Bootstrap's built-in JS.
  • Keyboard Interaction: Use semantic buttons. They support Tab, Enter, and Space by default.
  • Focus Order: Maintain logical tab order; don't auto-shift focus unless necessary.
  • Heading Semantics: Match <h2>, <h3>, etc. to your page’s outline for proper structure.
  • Decorative Icons: If used, apply aria-hidden="true" so screen readers skip them.
  • Best Practices: Refer to the ARIA Authoring Practices Guide for advanced behavior patterns.
