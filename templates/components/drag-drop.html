Drag and Drop

Drag and drop components allow users to reorder, move, or organize content by selecting items and placing them in a new location. Common use cases include sortable lists, file uploads, and interactive categorization tools. These interactions should provide a clear visual experience and alternative methods for users who cannot operate a mouse or perform drag-based gestures.

Below is a list of WCAG success criteria requirements and accessibility considerations:

  • Proper Use of ARIA Roles and Properties: Implement appropriate ARIA attributes for draggable items and drop targets (e.g., aria-grabbed, aria-dropeffect) to clearly indicate their state. 4.1.2 Name, Role, Value
  • Keyboard Navigation: Ensure drag and drop operations are fully operable via keyboard, allowing users to initiate, move, and drop items without relying on a mouse. 2.1.1 Keyboard
  • Dynamic Feedback and State Announcements: Utilize ARIA live regions to announce drag start, movement, and drop events, providing real-time feedback for state changes. 4.1.2 Name, Role, Value
  • Clear Instructions for Interaction: Provide text-based instructions that clearly explain how to perform drag and drop operations, ensuring users understand the process. 1.3.1 Info and Relationships
  • Focus Visibility: Ensure that focus indicators remain visible on draggable elements and drop zones throughout the interaction. 2.4.7 Focus Visible
  • ARIA Drag and Drop Best Practices: Follow the current W3C recommendations for drag and drop states and properties to ensure consistent behavior and accessibility. ARIA States and Properties, ARIA State: Drag and Drop

This example demonstrates an accessible drag-and-drop pattern that works seamlessly for mouse, touch, and keyboard users. Each list item is made draggable and includes a grip icon to visually indicate its functionality. Users can click and drag items with a mouse or use keyboard controls—pressing Enter or Space to pick up an item and using the Arrow keys to reposition it. Additionally, live region announcements ensure that state changes are clearly communicated to users relying on assistive technologies.

Example Explanation

HTML

Each list item is given draggable="true", a role of listitem, and a tabindex so that keyboard users can focus on them. A small SVG grip icon indicates that the item is draggable.

CSS

The styling uses a "grab" cursor by default and a "grabbing" cursor when the item is being dragged. Focus styles ensure that keyboard users can easily see the selected item. Additional classes (e.g., .dragging and .dragover) visually indicate different interaction states.

JavaScript

Mouse/Pointer: Standard drag events allow items to be moved using a mouse or touch (via pointer events).

Keyboard: When a list item is focused, pressing Enter or Space toggles the "keyboard drag" mode. While in drag mode, the Up and Down arrow keys reposition the item within the list. An ARIA live region announces each action, ensuring that users with assistive technologies are informed of every change.

Screen Reader Compatibility Note

During testing, we discovered that keyboard functionality for screen reader users (particularly with NVDA and JAWS) was blocked due to the screen reader intercepting keypresses like Space and Arrow keys. This prevented users from interacting with the component using the keyboard while the screen reader was active.

To resolve this, we applied role="application" to the drag-and-drop region. This instructs screen readers to suspend their virtual cursor mode and hand control of the keyboard back to the web application. As a result, the intended keyboard interactions—Space to enter drag mode, Arrow keys to reorder items, and Escape to cancel—now function correctly for screen reader users.

Important: Use role="application" carefully and only on the drag-and-drop container. Be sure to provide clear screen reader instructions (e.g., using an aria-describedby hint or persistent live region) and test thoroughly across NVDA, JAWS, and VoiceOver to avoid breaking screen reader navigation elsewhere on the page.

Drag and Drop Roles, Properties, States, and Tabindex Attributes

Role Attribute Element Usage
Draggable Item draggable="true" <li> (or similar) Enables the element to be moved using drag and drop interactions.
Draggable State aria-grabbed="false" <li> Indicates whether an element is currently grabbed (true) or not (false) during a drag operation.
Drop Target aria-dropeffect="move" <ul> or container element Indicates the intended drag action (e.g., move) when an element is dropped on the target.
Keyboard Focus tabindex="0" <li> (or similar) Makes the element focusable via keyboard, allowing users to initiate drag and drop using keyboard interactions.
Dynamic Announcements aria-live="assertive" <div> Provides real-time feedback to assistive technologies by announcing state changes during drag and drop operations.

When designing an accessible drag and drop pattern in Figma, it's critical to ensure that every visual element and interaction state meets WCAG 2.1 requirements. This means paying close attention to both text and non-text contrast, and clearly defining each state—static, dragging, drop, and focus—to support all users, including keyboard-only users and those with low vision.

  • Contrast for Text and Non-Text: Ensure that all text labels, instructions, borders, and focus indicators meet WCAG 1.4.3 (color contrast for text) and 1.4.11 (non-text contrast) requirements. This applies to every state of the drag and drop interaction.
  • Display All Interaction States: In your Figma prototypes, clearly design and document the static state, dragging state, drop state, and focus state. Each state should have distinct, high-contrast visual cues so that users can easily tell which state is active.
  • Clear Visual Feedback: Use visible indicators such as a contrasting grip icon, bold borders, or background changes to signal when an item is being dragged or is a drop target.
  • Keyboard Focus: Make sure that all draggable items and drop zones are fully accessible via keyboard. This includes having a visible and high-contrast focus indicator that meets both text and non-text contrast standards.
  • Collaborate with Content Leads and Auditors: Work closely with content leads and auditors to ensure that the drag and drop patterns reflect all requirements and accessibility best practices.

When auditing a drag and drop pattern in a Figma design, please review the designer’s implementation against the content leads’ documentation to ensure alignment and consistency. Evaluate the design for clear representation of every interaction state—including static, dragging, drop target, and keyboard focus—and verify that all elements meet the required contrast standards (WCAG 1.4.3 for text and 1.4.11 for non-text elements). Flag any issues for the designer to address, then annotate specific requirements for the developer handoff.

When auditing the drag and drop pattern, please consider the following accessibility issues to flag for review:

  • Visual Clarity of Draggable Items: Ensure that items are clearly marked as draggable with a visible grip icon and that their various states (static, dragging, drop target) are visually distinct.
  • Contrast and Focus Visibility: Confirm that all interaction states, including focus and active drag states, meet WCAG 1.4.3 and 1.4.11 contrast requirements, providing clear visual feedback.
  • Keyboard Interaction Representation: Verify that the design illustrates keyboard interactions (e.g., using Enter/Space to pick up/drop and arrow keys to reposition) and that these states are distinctly highlighted.
  • Dynamic State Feedback: Ensure that the design shows how dynamic state changes (via ARIA live regions) are communicated to users during drag operations.
  • Consistency Across Components: Check that the drag and drop pattern is applied uniformly throughout the design, with consistent styling and interaction cues.

Drag and Drop Pattern

Developer

Draggable Items

  • Elements must have draggable="true" and be assigned appropriate ARIA roles (e.g., role="listitem"), with a visible grip icon indicating movability.
  • Include aria-grabbed to reflect whether the element is currently being dragged.
  • Ensure items are focusable via tabindex="0" to support keyboard navigation.

Drop Targets

  • Drop zones should be clearly indicated and use ARIA states to reflect when an item is hovering over them.
  • Implement ARIA live regions to announce drag start, movement, and drop events for dynamic feedback.

Keyboard Interaction

  • Implement keyboard controls so that users can press Enter/Space to pick up or drop an item and use the Arrow keys to reposition it.
  • Ensure focus indicators are prominent and meet contrast requirements.

  • Draggable Elements: Ensure all draggable items have draggable="true", an appropriate role (e.g., listitem), and tabindex="0" to allow keyboard focus.
  • ARIA Grabbing State: Use aria-grabbed to dynamically indicate whether an element is currently being dragged (true) or not (false).
  • Drop Target Indicators: Clearly define drop zones with visual cues, and update ARIA attributes (or use programmatic announcements) when an item hovers over a valid drop target.
  • Live Region Announcements: Implement ARIA live regions (e.g., aria-live="assertive") to announce when an item is picked up, moved, or dropped.
  • Keyboard Interaction: Add keyboard event handlers so that users can press Enter/Space to pick up or drop items and use the arrow keys to reposition them. Ensure focus management is robust throughout the interaction.
  • Test Across Input Modalities: Verify that the drag and drop functionality works seamlessly with screen readers, keyboard-only navigation, mouse, and touch interactions.
  • Use ARIA Roles and Properties: Apply appropriate ARIA roles and properties to both draggable items and drop zones to ensure their states are programmatically determinable. For example, reference aria-grabbed for draggable items.
  • Manage Focus: Ensure focus indicators remain visible during drag and drop operations and that focus moves logically—from the draggable item when picked up to its new position when dropped.
  • Screen Reader Testing: Conduct thorough testing with screen readers to confirm that dynamic changes are announced correctly and that all instructions are clear.
  • Follow Best Practices: Align your implementation with current ARIA authoring practices and W3C recommendations for drag and drop interactions. Document your approach and test with all input methods to ensure robust accessibility.
