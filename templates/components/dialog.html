Dialog

  • Proper Use of ARIA Roles and Properties: Implement role="dialog" on dialog components to ensure they are recognized as modal dialogues by assistive technologies, which aids in accessibility management. ARIA: Dialog Role
  • Focus Management: Ensure that focus is managed correctly by setting initial focus on the first focusable element or the dialog container, and trapping focus within the dialog while it is open. This should adhere to the WCAG 2.4.3 Focus Order to ensure a logical sequence. 2.4.3 Focus Order
  • Keyboard Interaction: Dialogs must be fully operable via keyboard, including the ability to open, navigate within, and close them using standard keyboard commands, ensuring compliance with WCAG 2.1.1 Keyboard Accessible. ARIA: Dialog Example
  • ARIA Attributes: Use aria-modal="true" to enhance dialog modality, and aria-labelledby for associating the dialog with its label, ensuring the role, name, and value are correctly used as per WCAG 4.1.2. 4.1.2 Name, Role, Value
  • Dismissible: Ensure dialogs covering other content are dismissible using the ESC key, aligning with accessible design practices and enhancing user control. 1.4.13 Content on Hover or Focus

  • Trigger and Close Button Design: Display the dialog trigger button in its static state and after activation. Ensure that the close and cancel buttons within the dialog are clearly visible and designed consistently with other buttons in the application. This helps with user familiarity and expectation management.
  • Dialog Visibility: Design the dialog to stand out visually from other content. Use layering and shadows to indicate that the dialog is active and the content behind it is inactive. This visual differentiation helps users focus on the dialog content without distractions.
  • Content Accessibility: The content beneath the dialog should be made inaccessible while the dialog is open, typically by using the inert attribute or equivalent techniques. This prevents users from interacting with background content and ensures the dialog remains the focus of interaction.
  • Consistent Interaction: Include visible mechanisms such as a 'cancel' or 'close' button within the dialog to allow easy dismissal. These controls should be prominently placed and easy to interact with, facilitating accessibility for users with limited dexterity or visibility.
  • Layering and Focus Management: Show examples of the dialog in an open state and ensure it is layered appropriately over other content. Describe how focus is managed, e.g., focus should shift to the dialog immediately when opened and should trap focus until the dialog is dismissed. This prevents loss of focus and disorientation for users relying on keyboard navigation.
  • Visual Consistency: Maintain consistency in the design of dialog elements with other UI components. Use familiar visual cues such as borders, spacing, and typography to enhance readability and usability.
  • Non-reliance on Color Alone: Ensure that critical information in the dialog is not conveyed by color alone. Use text labels, icons, or patterns to communicate important messages, making the dialog accessible to users with color vision deficiencies.

When auditing a Dialog Modal component within a Figma design, please review the documentation provided by content leads and the designer's implementation to ensure consistency and alignment with accessibility standards.

During the audit of the Dialog Modal component in Figma, consider the following accessibility issues to flag any discrepancies for the designer's review:

  • Button Appearance: Verify that all buttons, including those to close or cancel the dialog, are visually identifiable as buttons. Check for visual consistency with other interactive elements within the application.
  • Contrast Requirements: Ensure that buttons and text within the dialog meet the minimum contrast ratio requirement of 4.5:1 for normal text to guarantee legibility for users with visual impairments.
  • Dialog Closure Mechanisms: Confirm that there are mechanisms, such as a close or cancel button, clearly provided to allow users to dismiss the dialog. These should be easily accessible and operable.
  • Dialog State Visibility: Check that the modal dialog is available in both its closed and open states within the design. If the open state is not displayed, make a note to the designer to include this in their Figma frames.
  • Consistency and Usability: Review the dialog for consistent use of UI components like buttons and check that all functional elements such as close mechanisms are in place and conform to recognized patterns.
  • Non-reliance on Color: Ensure that no critical information within the dialog is conveyed by color alone. All necessary information should be accessible through text or other non-color dependent means.

Dialog (Modal)

Dialog name, role and value:

  • Must have role="dialog" on the parent container.
  • Must have aria-modal="true" on the parent container.
  • Should have an H1 element with a unique ID attribute. "_____" should be the <h1>.
  • Must have an accessible name. It is preferable to use aria-labelledby equal to the H1's ID on the parent container instead of aria-label.

Dialog Trigger Button:

  • Must be a <button> or have role="button" and tabindex="0" with event handlers.
  • Must have an accessible name. The name should include a meaningful icon and additional sr-only text describing which glossary definition will be opened.
  • Must have aria-haspopup="dialog" on the triggering element or include screen-reader only text in the element indicating it will open a dialog.

Dialog Focus:

  • Must get focus when launched. Send focus to the container with the role="dialog" that has a tabindex="-1".
  • Must trap focus within the dialog while open.
  • Additionally, allow closing the dialog by pressing the Escape key (usability enhancement).
  • When the dialog is closed, return focus to the element which triggered the dialog, unless the element has been removed from the page, in which case place focus on the next logical element.

Dialog (Non-modal)

For Non-modal dialogs recommend the same as Modal Dialogs except do not recommend aria-modal="true" and do not recommend a focus trap. But if the non-modal dialog covers any meaningful content when open please recommend that it collapse when focus leaves.
