Button Vs. Link

A button <button> performs an action. (Open Disclosure, Play Video, Launch Dialog)

A link <a> navigates. (In-page anchor link, internal/external URL)

A button is a fundamental interactive element used to trigger actions, such as submitting a form, opening a modal, or toggling visibility of content. Buttons should be clearly labeled, operable via keyboard, and announced properly to assistive technologies. Accessible buttons ensure users can easily understand and activate controls regardless of input method.

Below is a list of WCAG success criteria requirements and accessibility considerations:

  • Button Semantics: Use the <button> tag for button elements. If another element is used to create a button, add role="button" and JavaScript to replicate standard button functionality. ARIA: Button Role
  • Accessible Names: Ensure buttons have accessible names, which can be provided by inner text or by attributes like aria-label if the button contains only an icon or an image. 4.1.2 Name, Role, Value
  • Indicating Button States: Clearly indicate button states (e.g., disabled, active, current) visually and programmatically. Disabled buttons should have aria-disabled="true" and active buttons should indicate their state with aria-pressed for toggle buttons or aria-current for currently selected buttons. 4.1.2 Name, Role, Value
    • ARIA: aria-disabled
    • WAI-ARIA Authoring Practices 1.1: Button Examples
  • Dialog Opening Function: Clearly indicate the function of the dialog opening button with an aria-haspopup="dialog" attribute or by adding sr-only text "opens dialog" to warn users that activating the button will open a dialog window and send focus to the new window. a href="https://www.w3.org/WAI/WCAG21/Understanding/name-role-value.html" target="_blank">4.1.2 Name, Role, Value
  • Button Text Contrast: Text on buttons must have at least a 4.5:1 contrast ratio against their background for regular sized text and 3:1 for large size text. 1.4.3 Contrast (Minimum)
  • Button Contrast Usability: buttons should be visually distinguishable from their background with a clear border or background color. 1.4.11 Non-text Contrast
  • Image Buttons: Use <input type="image"> for image buttons, ensuring to provide a meaningful alt attribute that describes the button's action. 1.4.11 Non-text Contrast 1.1.1 Non-text Content
  • Ensuring Keyboard Accessibility: Buttons must be fully functional using a keyboard alone, including being focusable and interactive through keyboard events like Enter or Space keys. 2.1.1 Keyboard

Button States

Button states are important, not just button styling! If you are only toggling classes to visually manage state of your components, you are likely not appropriately conveying that state to users of assistive technologies.

Accessibility Considerations for Button Design in Figma

  • Distinguishable Buttons: Design buttons to be clearly distinguishable from links or non-interactive elements. Use shape, elevation, or texture in addition to color.
  • Descriptive Names for Action: Whether using text or icons, ensure the purpose of the button is clear and descriptive. Avoid ambiguity to make the interface intuitive for all users, including those using assistive technologies.
  • Contrast and Visibility: Ensure sufficient contrast between button icons (or text) and their background, adhering to a minimum contrast ratio of 4.5:1 for text/icons to background. For larger text/icons, a minimum ratio of 3:1 is required.
  • State Design Examples: Provide clear design examples for different button states including:
    • Default state: How the button appears normally.
    • Focus state: Visually distinct changes when the button is focused, which is crucial for keyboard navigation.
    • Disabled state: Indications that the button is not interactive, which could include lower opacity or a change in color that still meets contrast ratios.
    • Note: While hover states enhance user experience, focus and disabled states are essential for accessibility.
  • Icons and Images on Buttons: When using icons or images on buttons, ensure that text alternatives are provided. This could be in the form of visible text on the button or through documentation for auditors and developers to implement alt text or aria-label attributes, ensuring the buttonâ€™s function is clear to all users, including those using screen readers.

Accessibility Considerations for Implementing Buttons in HTML

For developers translating Figma designs into accessible HTML, adhering to WCAG criteria 1.3.1 (Info and Relationships) and 4.1.2 (Name, Role, Value) is crucial, especially for buttons, including those with icons:

  • Semantic HTML: Utilize the <button> element for buttons to ensure they are accessible and their role is understood by assistive technologies. Avoid using <div> or <span> for interactive elements unless necessary.
  • Accessible Name for Icon Buttons: When a button is represented by an icon without visible text, use aria-label or aria-labelledby to provide an accessible name that describes the button's action, ensuring it is comprehensible to screen reader users.
  • Interactive States: Define clear visual and programmable states (hover, active, focus, disabled) for buttons using CSS and ARIA attributes to communicate the state changes to all users, including those using assistive technologies.
  • Contrast and Visibility: Ensure that buttons meet the minimum contrast ratio requirements against their background. Use CSS to achieve a contrast ratio of at least 4.5:1 for text and icons to background. Larger elements require a ratio of at least 3:1.
  • Focus Management: Implement proper focus management to ensure keyboard users can navigate through interactive elements seamlessly. Use tabindex where necessary and ensure custom buttons made from non-semantic elements are fully keyboard accessible.
  • Use of ARIA: Apply ARIA roles and properties judiciously to enhance accessibility when native HTML does not suffice. For instance, aria-pressed for toggle buttons to indicate their state.

By focusing on these development practices, developers can ensure that the buttons in the web interface are accessible, providing a better user experience for everyone, including those relying on assistive technologies.
