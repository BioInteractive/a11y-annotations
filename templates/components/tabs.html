Tabs

  • Proper Use of ARIA Roles and Properties: Implement role="tablist", role="tab", and role="tabpanel" to structure the tab interface, ensuring tabs and their content are properly recognized by assistive technologies. ARIA: Tab Panel
  • Keyboard Accessibility: Ensure all tabs are accessible via keyboard, following the pattern that tabs must be <button> in focus order, to comply with 2.1.1 Keyboard
  • ARIA Association: Use aria-controls to connect tabs with their corresponding tabpanel, and aria-selected="true" on the active tab, ensuring clear indication of active vs. inactive tabs.
  • Content Brevity: Keep tab labels concise to facilitate ease of understanding and operation, enhancing usability for all users, especially those with cognitive limitations.

Role, Property, State, and Tabindex Attributes

Role Attribute Element Usage
tablist div Indicates that the element serves as a container for a set of tabs.
aria-labelledby="ID_REFERENCE" div Provides a label that describes the purpose of the set of tabs.
tab button
  • Indicates the element serves as a tab control.
  • Provides a title for its associated tabpanel.
aria-selected="true" button
  • Indicates the tab control is activated and its associated panel is displayed.
  • Set when a user activates a tab.
  • Does not change when focus moves in the tablist.
aria-selected="false" button
  • Indicates the tab control is not active and its associated panel is NOT displayed.
  • Set for all tab elements in the tab set except the active tab; the one associated with the currently displayed panel.
tabindex="-1" button
  • Removes the element from the page Tab sequence.
  • Set when a tab is not selected so that only the selected (active) tab is in the page Tab sequence.
  • Since an HTML button element is used for the tab, it is not necessary to set tabindex="0" on the selected (active) tab element.
  • This approach to managing focus is described in the section on Managing Focus Within Components Using a Roving tabindex.
aria-controls="ID_REFERENCE" button Refers to the tabpanel element associated with the tab.
tabpanel div
  • Indicates the element serves as a container for tab panel content.
  • Is hidden unless its associated tab control is activated.
aria-labelledby="ID_REFERENCE" div
  • Refers to the tab element that controls the panel.
  • Provides an accessible name for the tab panel.

  • Clear Tab Indication: Ensure tabs are visually distinct and easily identifiable from other interactive elements. Use visual indicators like borders, backgrounds, or underlines to denote active and inactive tabs.
  • Icon Usage: If icons are used within tabs, ensure they are clear and help convey the function or content of the tab. Icons should be easily recognizable and consistent throughout the design.
  • Contrast and Legibility: Maintain a high color contrast between the text of the tabs and their backgrounds to ensure readability for all users, including those with visual impairments. Follow the WCAG guidelines for text and background contrast ratios.
  • Interactive Element Design: Design tab controls to look and behave like interactive elements. They should be easily clickable, with a large enough hit area, and visually respond to hover and focus states.
  • Content Organization: Design tabs to clearly organize content without overwhelming users. Each tab should correspond to content that is related and relevant, reducing the need for excessive navigation.
  • Consistency Across Views: Ensure that tabs have a consistent design and interaction model across different screens or sections of the application to prevent confusion and enhance usability.

When auditing a Tabs component within a Figma design, it is crucial to review the content leads documentation and the designer's implementation for alignment and consistency. This ensures that the design meets accessibility standards before moving to development.

During the audit of the Tabs component in Figma, consider the following accessibility issues to flag for the designer's review:

  • Focus Management: Ensure that focus order is designed to be logical and consistent, following the WCAG 2.4.3 Focus Order guidelines. Tabs should receive focus in the order they appear and maintain focus within the tab content when selected.
  • Visual Indicators: Ensure that there is a clear visual indicator of the active tab, helping users understand which tab content is currently visible.
  • Contrast and Legibility: Review the color contrast of text and interactive elements within the tabs to ensure they meet WCAG 1.4.3 Contrast (Minimum) requirements.
  • Content Clarity: Examine the clarity and conciseness of text used within the tabs, ensuring it is free from jargon and easily understandable.

Tabs with Manual Activation

  • Must have role="tablist" on containing element
  • Must have aria-labelledby equal to the unique ID of the heading element giving the tablist an accessible name of "..."
  • Must have role="tab" on each tab button
  • Place focus on the first active tab when navigating with the keyboard Tab key.
  • Must have tabindex="-1" on all non-active tabs to allow for arrow roving tabindex navigation within the tablist.
  • Must have toggling aria-selected="true/false" to active/non-active tab buttons
  • Add aria-controls to each button equal to the unique ID of the tabpanel the button controls
  • Must have role="tabpanel" to each panel and an aria-labelledby equal to the unique ID of the button that controls the panel.
