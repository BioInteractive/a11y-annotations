  • Provide Text Alternatives: Use aria-valuenow, aria-valuemin, aria-valuemax, and aria-label attributes to convey the current progress to assistive technologies. WCAG 1.1.1 Non-text Content
  • Ensure Sufficient Contrast: Text or information within the progress bar should have a sufficient contrast ratio against its background (at least 4.5:1). WCAG 1.4.3 Contrast (Minimum)
  • Make It Keyboard Accessible: Ensure the progress bar is operable through a keyboard interface and that focus can be moved to the progress bar. WCAG 2.1.1 Keyboard
  • Provide Clear Instructions: Clearly indicate the purpose and usage of the progress bar, making it understandable for all users. WCAG 3.3.2 Labels or Instructions
  • Use Appropriate ARIA Roles and Properties: Implement role="progressbar" and use aria-valuenow, aria-valuemin, and aria-valuemax attributes to provide the current progress value. WCAG 4.1.2 Name, Role, Value
  • Avoid Creating a Seizure Risk: Ensure the progress bar does not flash more than three times in any one-second period. WCAG 2.3.1 Three Flashes or Below Threshold

Role, Property, State and Name Attributes

Role Attribute Element Usage
progressbar div Defines the element as a progress bar to assistive technologies.
aria-label="Progress" div Provides an accessible name for the progress bar.
aria-valuemin="0" div Indicates the minimum value of the progress bar. Typically set to "0".
aria-valuemax="100" div Indicates the maximum value of the progress bar. Typically set to "100".
aria-valuenow="value" div Represents the current progress value. The value should be dynamically updated as the progress changes.
aria-valuetext="text" div Provides a user-friendly representation of the current value, particularly useful when the progress bar does not represent a numeric value.

  • Visual Clarity: Ensure the progress bar is easily distinguishable, using clear labels and distinct colors to indicate progress.
  • State Indicators: Design the progress bar to show clear indicators of current progress, start, and end points.
  • Consistent Association: Maintain a clear visual relationship between the progress bar and the content it is tracking to help users understand the progress.
  • Contrast and Legibility: Use sufficient color contrast for the progress bar and any accompanying text to ensure legibility and visibility for all users.
  • Responsive Design: Ensure the progress bar is designed to be responsive and functional on different screen sizes and devices.
  • Text Alternatives: Provide text alternatives for the progress bar to ensure that users who rely on screen readers can access the same information.
  • Animations and Transitions: If using animations or transitions, ensure they are smooth and do not cause distractions or issues for users with motion sensitivity.
  • Accessible Feedback: Design for clear and accessible feedback to users when progress changes, including any error states or completion messages.

When auditing the Progress Bar component within a Figma design please consider the following accessibility issues to flag for the designer to review:

  • Visual Clarity of Progress Bar: Ensure the progress bar is clearly designed and easily distinguishable from other content. It should be visually clear what the progress bar is indicating.
  • State Indicators: Verify if there are clear indicators showing the current progress, start, and end points. Ensure these indicators are visually intuitive.
  • Consistent Association: Check that the progress bar is clearly associated with the task or content it is tracking, with visual cues to indicate this relationship.
  • Consistency of Design: Assess the consistency in the design of all progress bar components throughout the project to ensure a uniform user experience.
  • Legibility: Confirm that text within or near the progress bar is legible, with adequate font size and color contrast.
  • Accessible Feedback: Ensure that there is accessible feedback for users when progress changes. This can include ARIA live regions, alerts, or messages that inform the user of the result of their input.
  • Responsive Design: Verify that the progress bar is designed to be responsive and functional on different screen sizes and devices.
  • Animations and Transitions: If using animations or transitions, ensure they are smooth and do not cause distractions or issues for users with motion sensitivity.

  • Implement role="progressbar" on the progress bar element.
  • Must have aria-valuemin, aria-valuemax, and aria-valuenow attributes to indicate the range and current value of the progress bar.
  • Ensure the progress bar is not operable via keyboard, as it is a read-only indicator.
  • Provide text alternatives for the progress bar value, if applicable, to ensure accessibility for screen reader users.
  • Include aria-live="polite" on dynamic updates to ensure screen readers announce changes.

  • Use ARIA Roles: Apply role="progressbar" to the progress bar element to make it recognizable as a progress indicator to assistive technologies. This role ensures that screen readers can identify and announce the progress bar.
  • Set ARIA Properties: Implement aria-valuemin, aria-valuemax, and aria-valuenow to indicate the minimum value, maximum value, and current value of the progress bar, respectively. This provides context and updates to screen reader users about the progress.
  • Announce Changes: Use aria-live="polite" to ensure that updates to the progress bar are announced to screen readers without interrupting the user. This attribute is essential for providing real-time feedback about ongoing processes.
  • Ensure Visual Indicators: Provide a visual representation of the progress bar with sufficient contrast and a clear indication of progress. This is critical for users with visual impairments.
  • Provide Text Alternatives: Offer text alternatives for the progress bar value, ensuring that the progress status is accessible to screen reader users. This can be done by providing additional text descriptions or tooltips.
  • Focus Management: Manage focus appropriately when updates to the progress bar occur. Ensure that focus is not unintentionally moved, maintaining a logical and intuitive focus order.
  • Testing and Validation: Regularly test the progress bar with assistive technologies and keyboard navigation to ensure it behaves as expected and is fully accessible. This includes verifying the correct announcement of progress changes by screen readers.
