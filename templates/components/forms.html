Forms

  • 1.3.1 Info and Relationships Information, structure, and relationships conveyed through presentation can be programmatically determined or are available in text, ensuring that form fields and their labels are properly associated.
  • 1.3.5 Identify Input Purpose The purpose of each input field collecting information about the user can be programmatically determined when the field serves a common purpose.
  • 2.4.6 Headings and Labels Headings and labels describe the topic or purpose, making it easier for users to find content and complete forms.
  • 3.2.2 On Input Changing the setting of any user interface component does not automatically cause a change of context unless the user has been advised of the behavior before using the component.
  • 3.3.1 Error Identification If an input error is automatically detected, the item that is in error is identified and the error is described to the user in text.
  • 3.3.2 Labels or Instructions Labels or instructions are provided when content requires user input.
  • 3.3.3 Error Suggestion If an input error is automatically detected and suggestions for correction are known, then the suggestions are provided to the user, unless it would jeopardize the security or purpose of the content.
  • 3.3.4 Error Prevention (Legal, Financial, Data) For Web pages that cause legal commitments or financial transactions for the user to occur, that modify or delete user-controllable data in data storage systems, or that submit user test responses, the user is able to reverse, correct, or confirm the action.
  • 4.1.2 Name, Role, Value For all user interface components (including but not limited to: form elements, links, and components generated by scripts), the name and role can be programmatically determined; states, properties, and values that can be set by the user can be programmatically set; and notification of changes to these items is available to user agents, including assistive technologies.
  • 4.1.3 Status Messages Status messages can be programmatically determined through role or properties such that they can be presented to the user by assistive technologies without receiving focus, especially important for conveying success or error messages after form submission.

Designing accessible forms is crucial for ensuring all users can interact with content effectively. Follow these guidelines to create forms in Figma that meet accessibility standards:

  1. Label All Form Elements Clearly: Ensure every form field has a label that describes its purpose. Use text labels that are visually connected to their corresponding form controls to make it clear what each input is for.
  2. Use Placeholder Text Appropriately: While placeholder text can provide hints or examples, it should not be used as the only method of providing instructions. Placeholder text disappears when a user starts typing, which may confuse users who need more context.
  3. Design Visual Cues for Required Fields: Clearly indicate which fields are required using visual cues, such as asterisks (*) next to the labels of required fields. Include instructions at the beginning of the form explaining what the cues mean.
  4. Ensure Sufficient Color Contrast: Design form elements with sufficient contrast between text labels, input text, and background colors to meet or exceed the minimum contrast ratios defined by WCAG guidelines.
  5. Create Descriptive Error Messages: Design error messages that name the field in error, give an explanation of error, and provide any required format information. Position error messages where they are easily noticed, such as next to the form field in question.
  6. Design for Different States: Include designs for various form states, such as hover, focus, error and success states, to provide feedback to users as they interact with the form.
  7. Provide Instructions for Complex Inputs: When designing forms that require data in a specific format (e.g., date fields), include instructions or examples to clarify the expected input format.
  8. Consider the Order of Form Elements: Organize form elements in a logical order that follows natural reading patterns, typically top to bottom and left to right, to ensure a smooth flow for users filling out the form.

Form Labels, Autocomplete, and Instructions

  • Form Controls must have a persistently visible <label> with a for/id association.
  • The <label> must identify if it is required with an (*) or the word (required).
  • Add appropriate autocomplete attributes to all fields requiring user specific information.
  • Form Instructions must be provided if the <label> identifies that it is a required field with an asterisk (*).
  • Provide instructions for context before the <form> "Fields with an asterisk (*) are required." or inside the <form> with an aria-labelledby association.

Form Validation and User Notification

  • Form Validation must be provided for all required form fields. Do not rely on HTML default error messaging.
  • Make Errors Prominent: Upon form submission or when focus is on the incorrectly filled-in field, errors should be made highly visible to the user.
  • Associate Error Messages: Utilize the aria-describedby attribute to programmatically associate each error message with the field in error. (The aria-describedby attribute should only be present when the error message is visible.)
  • Explicit Field Identification: Error messages should explicitly identify the field that is in error to remove any ambiguity. Best practice is to start with the field name, for example "First Name is required".
  • Format Suggestions: If the error is related to incorrect formatting, the error message should provide a suggestion on how to correct it.
  • Disable HTML5 Validation: Use the novalidate attribute on the form tag to disable HTML5's built-in validation, allowing for a more accessible custom validation process.
  • Use aria-invalid: Implement the aria-invalid="true" attribute on fields that have failed validation to inform assistive technologies that the field contains an error. (This attribute should be removed when the form field is not invalid.)
  • Use in-line error indication with clear descriptions for screen readers. Consider role="alert for errors.
  • Show descriptive validation messages with aria-live, ensuring it's in the DOM on page load.
  • Provide prominent success messages. Consider role="status on the container and ensure it's in the DOM on page load.

Developing accessible forms is essential for compliance with WCAG standards and ensuring usability for all users. Utilize the design and audit annotations to guide your development:

  1. Associate Labels with Form Controls: Use the <label> element for each form control. For better accessibility, ensure labels are programmatically associated with their corresponding input fields using the for attribute on the label that matches the id attribute on the input.
  2. Ensure Interactive Elements are Keyboard Accessible: Make sure all form elements can be accessed and used with a keyboard alone. This includes being able to navigate to and from each field, as well as submitting the form.
  3. Use Semantic HTML Where Possible: Use appropriate HTML5 input types (e.g., email, tel, date) to provide built-in validation and improve the user experience on various devices.
  4. Provide Clear Error Identification: Implement client-side validation that provides specific, actionable error messages. Errors should be programmatically associated with their fields using the aria-describedby attribute on the field that matches the id attribute on the error message.
  5. While in Error State: Fileds in error must have the aria-invalid="true" attribute and users focus should be directed to the first error if the form fails to submit.
  6. Ensure Forms Have Accessible Names: Use the <legend> element within <fieldset> or aria-labelledby to provide an accessible name for forms, especially if the form serves a specific purpose or is one of multiple forms on a page.
  7. Handle State Changes Accessibly: For dynamic forms that change state based on user input, ensure that these changes are announced to screen reader users via ARIA roles and properties, such as aria-live for status messages.
  8. Include ARIA Descriptions for Complex Inputs: For inputs that require specific formats or have complex validation rules, use ARIA attributes such as aria-describedby to reference additional descriptive text or error messages.
  9. Provide Clear Success Messages: Ensure success messages are immediately visible and clearly state the action was successful, using ARIA roles to make them accessible to screen reader users.

Labeling Controls

Persistently visible labels must be provided to identify all form controls, including text fields, checkboxes, radio buttons, and drop-down menus. Placeholder text is not a sufficient label becuase it disappears as soon as the user begins to type in the input field.

Visible labels must be close in proximity and must identify the field Labels by describing the purpose of the form control. If the field is a required field the label must identify it by either an asterisk (*) or the word (required) in the label.

In some instances an aria-label or aria-labelledby can also be used to identify form controls. This approach should only be used when the label of the control is clear from the surrounding content. For example if the search button has a well known magnifying glass icon or the search form button is labelled "search" then an additional visible label is not a hard requirment.

It is mandatory to provide autocomplete attributes to the fields on the form requesting information about the user. When autocomplete is assigned for input fields, suggestions are provided for the user. For some users when that does not happen it is difficult to complete the form.

Form Validation

If there are forms present that contain required input fields or input fields requiring specific data validation please ask the designer to provide you with examples of the form while in error state. The form error messages should be reviewed for color contrast, identify the field in error using the field name, and providing any required format information.

While HTML5 form validation offers a quick way to check for basic input errors, it falls short in several key areas of accessibility. Specifically, it does not identify the field in error using the field name, the error messages do not stay persistently visible, and it does not provide feedback about required formats for most field types. Additionally, the error messages are not programmatically associated with the field in error and often visually obscure other content, especially on smaller screen sizes. Client side error messaging is always preferred.

To address these issues, the following steps are recommended:

  1. Make Errors Prominent: Upon form submission or when focus is on the incorrectly filled-in field, errors should be made highly visible to the user.
  2. Associate Error Messages: Utilize the aria-describedby attribute to programmatically associate each error message with the field in error. (The aria-describedby attribute should only be present when the error message is visible.)
  3. Explicit Field Identification: Error messages should explicitly identify the field that is in error to remove any ambiguity. Best practice is to start with the field name, for example "First Name is required".
  4. Format Suggestions: If the error is related to incorrect formatting, the error message should provide a suggestion on how to correct it.
  5. Disable HTML5 Validation: Use the novalidate attribute on the form tag to disable HTML5's built-in validation, allowing for a more accessible custom validation process.
  6. Use aria-invalid: Implement the aria-invalid="true" attribute on fields that have failed validation to inform assistive technologies that the field contains an error. (This attribute should be removed when the form field is not invalid.)

By implementing these recommendations, you'll be taking a significant step toward making your forms more accessible and user-friendly.

User Notification

Users must be notified after submitting a form. please ask the designer to provide you with examples of the form while in error state and success states. The user notifications should be reviewed for color contrast, identifying the status of the form submission and if the user is provioded with details about the submission.

After users submit a form, it's essential to provide clear and accessible feedback about the results of their submission. This feedback can be categorized into two main types: in-line feedback and overall feedback.

In-line feedback is real-time feedback provided as users interact with the form controls. It helps users correct errors and understand the status of their input as they go along. Key considerations for in-line feedback include:

  • Error Indication: When users enter incorrect or incomplete information, errors should be visually and programmatically indicated near the associated form controls. You can use techniques like color changes, icons, or error messages next to the affected fields. Ensure these indications are perceivable and provide textual descriptions for screen reader users that are announced with role="alert", aria-live regions or role="status" On the container where the text notification is displayed. Please ensure these rules and live regions are in the DOM on page load.
  • Validation Messages: Use descriptive and concise validation messages to inform users about specific errors or formatting requirements. These messages should appear promptly when errors occur and disappear when the user corrects them. Utilize the aria-live attribute to make dynamic updates accessible to screen reader users.

Overall feedback is typically provided after the user submits the form and it has been processed. This feedback informs users about the success or failure of their submission and any further actions they may need to take. Key considerations for overall feedback include:

  • Success Message: If the form submission is successful, display a clear and positive message indicating success. Mention any next steps or actions if applicable, such as "Your information has been successfully submitted."
  • Error Message: If the form submission encounters errors, provide a detailed error message that explains what went wrong and how to correct it. Ensure that error messages are prominent, clear, and accessible to all users, including those using screen readers.
  • Dynamic Announcements: Make sure that both in-line and overall feedback is accessible to users with disabilities. Use appropriate ARIA attributes like aria-live and aria-atomic to ensure screen readers announce updates correctly.

Custom Controls

If you are unclear about a custom control element on a design ask the designer or stakeholder to provide you with detailed information about the function of the control. Knowing the intended function of the control should give you a better idea of how to recommend making it accessible in the development phase.

Custom controls in web forms offer a powerful way to improve accessibility and user experience. They allow you to design and implement form elements that meet the specific needs of your users while ensuring compliance with accessibility standards.

Key considerations to recommend to developers when creating custom controls include:

  • Semantic Markup: Use semantic HTML elements as a foundation for custom controls. This provides a clear structure and ensures compatibility with assistive technologies.
  • Keyboard Accessibility: Ensure that custom controls are fully operable using a keyboard. Implement keyboard event handlers to enable keyboard interaction.
  • Focus Management: Manage focus states carefully to provide visual and programmatic feedback to users. This includes focus styles and ARIA attributes.
  • ARIA Roles and States: Utilize ARIA roles and states to convey the purpose and state of custom controls to assistive technologies.
  • Testing and Feedback: Thoroughly test custom controls with various assistive technologies and gather user feedback to refine their design and functionality.

By creating custom controls with accessibility in mind, you can enhance the usability and inclusivity of your web forms, ensuring that all users can interact with them effectively.
