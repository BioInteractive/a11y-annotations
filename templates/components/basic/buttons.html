Buttons

  • Button Semantics: Use the <button> tag for button elements. If another element is used to create a button, add role="button" and JavaScript to replicate standard button functionality. ARIA: Button Role
  • Accessible Names: Ensure buttons have accessible names, which can be provided by inner text or by attributes like aria-label if the button contains only an icon or an image. 4.1.2 Name, Role, Value
  • Indicating Button States: Clearly indicate button states (e.g., disabled, active) visually and programmatically. Disabled buttons should have aria-disabled="true" and active buttons should indicate their state with aria-pressed for toggle buttons or aria-current="page" for navigation buttons.
    • ARIA: aria-disabled
    • WAI-ARIA Authoring Practices 1.1: Button Examples
  • Visual Design and Contrast: Design buttons with adequate color contrast ratios. Text or icons on buttons must have at least a 4.5:1 contrast ratio against their background, and buttons should be visually distinguishable from their background with a clear border or background color. 1.4.3 Contrast (Minimum)
  • Ensuring Keyboard Accessibility: Buttons must be fully functional using a keyboard alone, including being focusable and interactive through keyboard events like Enter or Space keys. 2.1.1 Keyboard
  • Image Buttons: Use <input type="image"> for image buttons, ensuring to provide a meaningful alt attribute that describes the button's action. 1.4.11 Non-text Contrast

Button States

Button states are important, not just button styling! If you are only toggling classes to visually manage state of your components, you are likely not appropriately conveying that state to users of assistive technologies.

  • Clear Button Identification: Design buttons to visually communicate their functionality, ensuring they are easily distinguishable from other elements like links. Buttons should look like buttons, implying clickability through design cues such as shadows, gradients, or shapes that suggest interactivity.
  • Contrast Ratios: Ensure text and outline colors on buttons meet a minimum contrast ratio of 4.5:1 against their background to comply with WCAG 2.1 AA guidelines for text and images of text (1.4.3 Contrast (Minimum)). This guideline ensures that text on buttons is legible to users with visual impairments.
  • Touch Target Size: Design buttons with a minimum touch target size of 44x44 pixels to meet WCAG 2.1 guidelines for target size (2.5.5), ensuring that buttons are easily clickable or tappable, enhancing usability for all users, including those with mobility impairments.
  • Functional Distinction: If the component navigates to another page, it should be designed as a link and look like a link; if it performs an action (e.g., submission, expansion), it should be designed and function as a button. This distinction helps users understand the action they are about to take.
  • State Design Examples: Provide clear design examples for different button states including:
    • Default state: How the button appears normally.
    • Focus state: Visually distinct changes when the button is focused, which is crucial for keyboard navigation.
    • Disabled state: Indications that the button is not interactive, which could include lower opacity or a change in color that still meets contrast ratios.
    • Note: While hover states enhance user experience, focus and disabled states are essential for accessibility.
  • Icons and Images on Buttons: When using icons or images on buttons, ensure that text alternatives are provided. This could be in the form of visible text on the button or through documentation for auditors and developers to implement alt text or aria-label attributes, ensuring the button’s function is clear to all users, including those using screen readers.

  • Color Contrast: Check that text and visual elements of the pagination component have sufficient color contrast ratios as per WCAG guidelines to ensure legibility for users with visual impairments.
  • Visual Clarity of Interactive Elements: Verify that interactive elements such as page numbers, 'Previous', and 'Next' buttons are visually distinguishable from non-interactive elements and clearly indicate their purpose.
  • Disabled State Indication: Review designs for disabled states (e.g., 'Previous' button on the first page) to ensure they are clearly indicated as non-interactive while remaining visible to the user.
  • Active State Indication: Ensure that the current page number is visually distinct from other page numbers to clearly indicate the current position in the pagination sequence.
  • Consistency Across Pages: Assess the consistency of pagination design across different sections or pages of the product to ensure a predictable navigation experience for users.
  • Icons and Visual Indicators: Examine the use of icons or visual indicators (like arrows or ellipses) for their simplicity, recognizability, and whether they are accompanied by clear, descriptive labels or implied visual meanings.
  • Hover and Focus State Designs: Check for the presence of hover and focus state designs for interactive elements, recognizing that actual state changes will be verified in development.
  • Design for Large Sets of Pages: For designs accommodating large sets of pages, ensure visual indicators for page skipping (e.g., ellipses) are clear and understandable.

Please refer to General > Structure > Buttons for detailed annotations.

  • Semantic HTML: Use the <button> tag for action buttons to ensure accessibility and native browser support for keyboard interactions. For links that navigate, use the <a> tag and style it accordingly. HTML5 Button Element
  • Accessible Button States: Implement and test different states for buttons including default, hover, focus, and disabled states. Ensure the focus state is visually distinct to aid keyboard navigation, and use aria-disabled="true" for disabled buttons to convey state to assistive technologies.
  • Color Contrast: Verify that the text and border color contrasts of buttons meet or exceed the WCAG AA level requirements for minimum contrast ratios, ensuring text is legible and buttons are distinguishable. WCAG 2.1 1.4.3 Contrast (Minimum)
  • Button Appearance: Code buttons to be clearly identifiable as clickable elements distinct from other components. Utilize CSS to replicate the visual design provided in Figma, including dimensions, colors, and typography that make buttons appear as interactive elements.
  • Icons and Images on Buttons: If a button includes icons or images, ensure they are implemented with alternative text descriptions for screen readers. Use alt text for images and aria-label or aria-labelledby for icons.
  • Keyboard Accessibility: Test buttons to confirm they are fully functional with keyboard-only controls. This includes being able to focus, activate, and navigate between buttons using keyboard inputs such as the Tab key and Enter/Spacebar keys.
