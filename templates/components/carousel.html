Carousel

HHMI discourages the use of carousels due to frequent accessibility issues, such as navigation difficulties, disrupted focus order, inadequate screen reader support, and challenges with timing and visibility. These issues often fail WCAG requirements and can hinder user experience. Below, we provide common alternative approaches to present content more inclusively.

Alternative Ways to Present Carousel Content

  • Static Images and Content: Display all images and content on the page to avoid auto-rotation issues and ensure visibility.
  • Tabbed Interfaces: Organize content into easily navigable tabs for better accessibility and structure. W3C: Tab
  • Collapsible Sections: Use expandable sections to let users control what they view. W3C: Accordions

Component Accessibility Requirements

  • Must provide a mechanism to pause, stop, or hide auto-advancing carousels. 2.2.2 Pause, Stop, Hide
  • Must have appropriate roles and attributes to identify the carousel, including aria-roledescription="carousel" and a unique name with aria-labelledby. 4.1.2 Name, Role, Value
  • Must ensure proper focus order when navigating carousel controls and slides. 2.4.3 Focus Order
  • Must provide accessible navigation for all interactive elements (e.g., buttons, dots) with proper keyboard support and focus visibility. 2.1.1 Keyboard
  • Must announce updates to slide content using an aria-live region to ensure screen readers are informed of changes. 4.1.2 Name, Role, Value
  • Must visually and programmatically indicate the current slide using aria-current="true" on the corresponding navigation dot or control. 4.1.2 Name, Role, Value
  • Decorative elements in the carousel, such as background images or icons, must be hidden from assistive technologies using aria-hidden="true". However, if the only elements within the carousel are images, they are considered meaningful and must include correct, descriptive alt text. 1.1.1 Non-text Content

Role Attribute Element Usage
region section
  • Role region is implied for any section element that has an accessible name.
  • Defines the carousel and its controls as a landmark region.
aria-roledescription="carousel" section
  • Informs assistive technologies to identify the element as a "carousel" rather than a "region."
  • Affects how the assistive technology renders the role but does not affect functionality, such as commands for navigating to landmark regions.
  • NOTE: The aria-roledescription value is defined using the language of the page and should be localized by the author.
aria-label="Highlighted television shows" section Provides a label that describes the content in the carousel region.
aria-live="off" div.carousel-items
  • Applied to a div element that contains all the slides.
  • Identifies the container element as a live region that is in the "off" state, meaning assistive technology users are not informed about changes to the region.
  • The live region is off when the carousel is automatically rotating.
aria-live="polite" div.carousel-items
  • Applied to a div element that contains all the slides.
  • Identifies the container element as a live region in the "polite" state, meaning assistive technology users are informed about changes to the region at the next available opportunity.
  • This causes screen readers to automatically announce the content of slides when the next and previous slide buttons are activated.
group div.carousel-item
  • Applied to each of the elements that contains the content of a single slide.
  • Enables assistive technology users to perceive the boundaries of a slide.
aria-roledescription="slide" div.carousel-item
  • Informs assistive technologies to identify the element as a "slide" rather than a "group."
  • Affects how the assistive technology renders the role but does not remove any assistive technology functions related to group elements.
  • NOTE: The aria-roledescription value is defined using the language of the page and should be localized by the author.
aria-label="N of 6" div.carousel-item
  • Provides each slide with a distinct label that helps the user understand which of the 6 slides is displayed.
  • Note: Normally, including set position and size information in an accessible name is not appropriate. An exception is helpful in this implementation because group elements do not support aria-setsize or aria-posinset.
aria-label="LABEL_STRING" button Defines the accessible name for the pause auto-rotation button and the next and previous slide buttons.
aria-controls="IDREF" button
  • Identifies the content on the page that the button controls.
  • Refers to the div that contains all the slides.

  • Slide Visibility: Design carousel slides with clear, legible typography and ensure adequate spacing to prevent content overlap or crowding.
  • Color Contrast: Ensure all text and interactive elements within the carousel meet WCAG contrast ratio guidelines for readability, especially for users with visual impairments. Pay extra attention if text is placed over images, ensuring sufficient contrast between the text and the background.
  • Consistent Navigation Controls: Maintain consistent design and placement for navigation controls (e.g., next/prev buttons, dots) across all slides to aid in usability and orientation. All controls must meet HHMI's target size standard of 44x44 pixels.
  • Current Slide Indication: Highlight the currently active slide using a non-color indicator such as bold text or an underline, and ensure contrast ratios meet at least 3:1 for visual distinction.
  • Interactive Feedback: Provide clear hover, focus, and current states for navigation controls and slide indicators to help users understand which elements are interactive. Ensure that all buttons, including meaningful icons, pass a contrast ratio of at least 3:1.
  • Controls for Operation: Include controls to operate the carousel without relying solely on dragging motion. For autoplaying carousels, ensure designs include a clearly labeled pause/play button that meets the target size and contrast requirements.
  • Image Usage: If images alone are used in the carousel, they must be meaningful and include descriptive alt text. If text overlays are used on images, designers must request alt text and ensure proper contrast between the text and the image background.
  • Labeling and Clarity: Ensure all carousel elements, including navigation controls, are clearly labeled with descriptive text to provide context for their functionality.
  • Customization Options: Provide controls for customizing auto-rotation timing and enabling users to pause, stop, or restart the carousel, with labels clearly communicating these options.

How to Test and Annotate

  • Color Contrast:
    • Test the color contrast of all text content on the slides, ensuring it meets WCAG requirements (minimum 4.5:1 for normal text, 3:1 for large text).
    • Test the color contrast of all interactive buttons or slide controls, including arrows, navigation dots, and the pause/play button (minimum 3:1 for non-text elements).
    • Verify the contrast of any graphical icons, ensuring they meet 3:1 contrast requirements in all states: static, hover, focus, and active.
  • Pause/Play Button: Ensure a pause/play button is included in the design if the carousel will be autoplaying. Confirm it is labeled clearly and meets the target size standard (44x44 pixels).
  • State Visibility:
    • Check that hover, focus, and active states are visually distinct for all interactive elements, such as buttons and navigation dots.
    • Verify that the current slide is visually indicated (e.g., with bold/underlined text or a clear active state).
  • General Accessibility Considerations:
    • Test that any text overlaid on images is legible and meets contrast requirements with the background.
    • Check that graphical icons are not the sole means of conveying information, and include text labels where necessary.
    • Point out if any decorative elements are incorrectly labeled as interactive or accessible.

Breadcrumb Navigation

  • Wrap the carousel in a role="region" container with aria-roledescription="carousel".
  • Add aria-labelledby to the container, referencing the ID of the heading labeling the carousel.
  • Wrap each slide in a role="group" container with aria-roledescription="slide".
  • Assign a unique id to each slide.
  • Add visually hidden text for slide numbering (e.g., "# of #") and use aria-labelledby to reference it.
  • Use list markup (<ul>) for navigation dots; ensure each dot is a <button>.
  • Set aria-current="true" on the dot representing the current slide.
  • Hide non-visible slides using visibility: hidden to ensure assistive tech ignores them.

As a developer, your goal is to ensure the carousel is fully operable and understandable by all users, including those navigating with assistive technologies. The markup must clearly identify the carousel's role, expose current slide state, and support navigation via keyboard.

Accessible Carousel Code Example

This example shows a simple two-slide carousel pattern with semantic structure and ARIA attributes to support screen reader users and keyboard interaction.

  
  <section id="carousel1" role="region" aria-roledescription="carousel" aria-labelledby="carouselHeading">
    <h2 id="carouselHeading">Featured Content</h2>

    <div class="carousel-slide" role="group" aria-roledescription="slide" aria-labelledby="slide1-label">
      <div id="slide1-label" class="visually-hidden">Slide 1 of 2</div>
      <p>Content of slide 1</p>
    </div>

    <div class="carousel-slide" role="group" aria-roledescription="slide" aria-labelledby="slide2-label" aria-hidden="true">
      <div id="slide2-label" class="visually-hidden">Slide 2 of 2</div>
      <p>Content of slide 2</p>
    </div>

    <div class="carousel-controls">
      <button aria-label="Previous slide"></button>
      <button aria-label="Next slide"></button>
    </div>

    <ul class="carousel-dots">
      <li><button aria-label="Go to slide 1" aria-current="true"></button></li>
      <li><button aria-label="Go to slide 2"></button></li>
    </ul>
  </section>

Tip: Always ensure dynamic changes are announced. See Dynamic Changes for details on live regions.

Key Developer Considerations

General Requirements

  • Wrap the entire carousel in a container with role="region" and aria-roledescription="carousel".
  • Reference a unique heading using aria-labelledby to label the carousel.
  • Use <ul> and <button> elements for navigation dots, and apply aria-current="true" on the active dot.
  • Ensure all interactive controls (prev/next, pause/play) meet color contrast, size, and focus requirements.

Single Visible Slide Carousels

  • Wrap each slide in a container with role="group" and aria-roledescription="slide".
  • Include hidden labels such as "Slide 1 of 2" using aria-labelledby or aria-label.
  • Hide non-visible slides using aria-hidden="true" or equivalent CSS techniques like visibility: hidden.

Multiple Visible Slide Carousels

  • Wrap the visible slide group in a container with role="list".
  • Use <li> for each visible slide, and apply aria-roledescription="slide".
  • Ensure visible slides are accessible in the tab order and clearly indicate the active one using aria-current or a visually hidden label.
