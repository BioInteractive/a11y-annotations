W3C APG Combo Box Patterns

Below is a list of different combo box patterns that demonstrate various implementations of the combo box design pattern:

  • Select-Only Combobox: A single-select combobox with no text input that is functionally similar to an HTML select element.
  • Editable Combobox with Both List and Inline Autocomplete: An editable combobox that demonstrates the autocomplete behavior known as list with inline autocomplete.
  • Editable Combobox with List Autocomplete: An editable combobox that demonstrates the autocomplete behavior known as list with manual selection.
  • Editable Combobox Without Autocomplete: An editable combobox that demonstrates the behavior associated with aria-autocomplete="none".
  • Editable Combobox with Grid Popup: An editable combobox that presents suggestions in a grid, enabling users to navigate descriptive information about each suggestion.
  • Date Picker Combobox: An editable date input combobox that opens a dialog containing a calendar grid and buttons for navigating by month and year.

Note: If you plan to implement a simple combobox, it is often better to use the native HTML <select> element for accessibility and ease of use. Here’s an example:

                      <label for="example-select">Choose an option:</label>
                      <select id="example-select">
                        <option>Option 1</option>
                        <option>Option 2</option>
                        <option>Option 3</option>
                      </select>

When auditing a combobox component in Figma, review its design elements for accessibility, focusing on visual compliance. Pay attention to how the combobox is structured and whether key accessibility features are visually present.

  • Color Contrast Compliance: Ensure that the text and background colors for the input field and dropdown list meet WCAG 2.1 AA contrast requirements (at least 4.5:1 for normal text and 3:1 for large text).
  • Visual Indication of State: Verify that there is a clear visual indicator for whether the combobox is expanded or collapsed, such as an icon or arrow that changes state.
  • Focus and Hover States: Check that both the input field and dropdown options have clearly defined focus and hover states, with visual changes like border, background color, or text styling.
  • Labeling and Instructions: Ensure the combobox has visible labels and instructions that are clear and provide guidance for users on how to interact with it.
  • Selection States: Confirm that the selected option is visually distinct in the dropdown, using color or bold text to differentiate it from other options.

Combobox

  • Use appropriate HTML markup: Implement the combobox using an <input> element with role="combobox" for the input field and aria-controls to associate it with the list of options. Use role="option" for each item within the dropdown list.
  • Provide persistently visible labels: Ensure the combobox is labeled using the <label> element, with a for attribute pointing to the input’s id, so that screen readers correctly announce the label.
  • Ensure keyboard accessibility: Users must be able to open and close the combobox using the Enter key, navigate through options using arrow keys, and make a selection with Enter. The Escape key should close the dropdown without selecting an option.
  • Manage focus state: Use aria-activedescendant to manage which option is currently focused as the user navigates through the list with the keyboard. Ensure that the focused option is visually distinct.
  • State management: Implement aria-expanded to indicate whether the combobox dropdown is open or closed, and ensure it toggles appropriately with user interaction.
  • Maintain color contrast: Ensure all text and background colors within the combobox, input field, and dropdown meet WCAG 2.1 AA contrast requirements (4.5:1 for normal text, 3:1 for large text).
  • Define hover and focus states: Ensure the combobox input and each dropdown option have clearly defined hover and focus states, with visual changes such as background or border color.
  • Implement correct selection states: Use aria-selected="true" to indicate the selected option in the dropdown, ensuring it is visually distinct from non-selected options.
  • Refer to ARIA Combobox Patterns: Follow the detailed guidance in the APG Combobox Patterns for best practices in accessibility.

  • Use ARIA Roles: The combobox input should have role="combobox", and the associated list of options should have role="listbox". Each option inside the list should use role="option".
  • ARIA Attributes for Autocomplete: If the combobox supports autocomplete, use aria-autocomplete to specify the type. Common values include list (for showing suggestions) and none (for no autocomplete).
  • Manage ARIA States: Use aria-expanded="true/false" on the input to indicate whether the options list is visible or hidden. This should toggle correctly when the combobox is opened or closed.
  • Link Input and Listbox: Ensure that the aria-controls attribute is applied to the combobox input, pointing to the ID of the associated listbox element. This lets assistive technologies know which list of options is controlled by the input.
  • Ensure Keyboard Navigation: Comboboxes should be fully operable via keyboard. Use the arrow keys to navigate through the list of options, Enter to select an option, and Escape to close the dropdown. The combobox should follow standard keyboard navigation patterns.
  • Focus Management: Properly manage focus by ensuring the input retains focus while navigating options with the keyboard. Once an option is selected, the focus should return to the input field or move as required by the interaction pattern.
  • Implement Optional Autocomplete: If the combobox offers inline autocomplete, dynamically update the input value with suggestions as the user types. This should be done in a way that still allows users to type in their own input.
  • Screen Reader Behavior: Ensure screen readers announce both the current option being navigated and the overall status of the combobox (e.g., when the dropdown opens or closes).
  • Test Across Devices: Ensure that the combobox is tested across a variety of devices, including mobile, as touchscreens might require different interactions for dropdown lists.
  • Follow ARIA Best Practices: For more information on accessible comboboxes, consult the WAI-ARIA Authoring Practices for Comboboxes.
