Footer

The footer is a structural component found at the bottom of a webpage that typically contains secondary navigation, legal links, contact information, or social media icons. It provides users with persistent access to important site-wide resources and should remain consistent across pages to support orientation and usability.

Below is a list of WCAG success criteria requirements and accessibility considerations:

  • Landmark Roles: Use <footer> or role="contentinfo" to semantically identify the footer region for assistive technologies. 1.3.1 Info and Relationships
  • Keyboard Navigation: Ensure all links and interactive elements in the footer are focusable and operable via keyboard. 2.1.1 Keyboard
  • Focus Order: Footer content should follow logical focus order at the end of the page content. 2.4.3 Focus Order
  • Text Contrast: Ensure all text in the footer meets WCAG 2.1 minimum contrast ratios. 1.4.3 Contrast (Minimum)

Note: Avoid hiding footer links behind expandable disclosures unless required for space. They should be visible and easily navigable, especially for screen reader and keyboard users.

  • Use Real Content: Design the footer with actual link text, headings, and disclaimers instead of placeholders to reflect the real content structure.
  • Visual Hierarchy: Organize content into logical groupings (e.g., links, social media, legal) with clear section headings and spacing.
  • Color and Contrast: Maintain minimum contrast ratios for text and background. Don’t rely on low-opacity text or soft grays that may fail WCAG.
  • Link Indicators: Ensure that footer links are visually identifiable and follow the site’s styling for interactive elements.
  • Mobile Considerations: Make sure the footer design is responsive and legible at smaller screen sizes, without stacking links too tightly or hiding content.

When auditing a footer component in Figma, confirm that its structure and styling support accessibility and semantic clarity. The footer must be distinguishable, well-labeled, and contain operable interactive content.

  • Landmark Role: Ensure the footer is styled as a semantic <footer> element or includes role="contentinfo".
  • Link Grouping: Group navigation links with appropriate headings or ARIA labels (e.g., aria-label="Footer navigation").
  • Color Contrast: Confirm that text and links meet minimum color contrast requirements against the background (4.5:1 for regular text, 3:1 for large text).
  • Focus and Hover States: Verify that links and interactive elements include clearly styled hover and focus indicators.
  • Visual Consistency: Check for responsive layout support and verify that links and text are legible at different zoom levels and screen sizes.

Footer

  • Use a <footer> tag or apply role="contentinfo" to the footer container.
  • Group navigation links inside a <nav aria-label="Footer navigation"> container.
  • Ensure all links are focusable and operable by keyboard. Use appropriate HTML anchor tags.
  • Maintain WCAG-compliant color contrast ratios for all text, icons, and background colors.
  • Clearly define hover and focus styles using borders, background changes, or underlines.

Footers support orientation and navigation across all pages. Use proper semantic markup to ensure the region is identifiable and accessible to assistive technologies.

Accessible Footer Code Example

This example shows a semantic <footer> with labeled navigation and social links.


<footer class="bg-dark text-white p-4 mt-auto">
  <nav>
    <ul class="list-unstyled d-flex flex-wrap gap-3 mb-3">
      <li><a href="/about" class="text-white">About Us</a></li>
      <li><a href="/privacy" class="text-white">Privacy Policy</a></li>
      <li><a href="/terms" class="text-white">Terms of Use</a></li>
    </ul>
  </nav>

  <div class="d-flex gap-3">
    <a href="https://twitter.com/example" class="text-white" aria-label="Follow us on Twitter">
      <i class="fab fa-twitter" aria-hidden="true"></i>
    </a>
    <a href="https://linkedin.com/company/example" class="text-white" aria-label="Follow us on LinkedIn">
      <i class="fab fa-linkedin" aria-hidden="true"></i>
    </a>
  </div>
</footer>

Tip: If you include multiple <footer> regions (e.g., one inside a modal), distinguish them with aria-label for clarity.

Key Developer Considerations

  • Semantic Structure: Use <footer> and role="contentinfo" for screen reader recognition.
  • Landmark Naming: Apply aria-label if more than one footer exists.
  • Keyboard Navigation: All links and buttons must be focusable and operable via keyboard.
  • Focus States: Ensure visible and high-contrast focus indicators on all interactive elements.
  • Group Links Semantically: Use <nav> with aria-label for grouped links.
  • External Links: Use aria-label to announce links that open in new tabs or leave the domain.
  • Responsive Layout: Ensure clean reflow at small screen sizes and 400% zoom.
