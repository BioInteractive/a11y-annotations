Alert Dialog

An alert dialog is a special type of modal dialog that interrupts the user's workflow to deliver critical information or confirm a destructive action. Unlike standard dialogs, alert dialogs require immediate attention and must be acknowledged before the user can return to the main interface. They must be designed and implemented to support assistive technologies and keyboard navigation in strict conformance with WCAG 2.1.

Below is a list of WCAG success criteria requirements and accessibility considerations specific to alert dialogs:

  • Correct ARIA Role: Use role="alertdialog" instead of dialog to convey the urgency of the content and its interruptive nature. ARIA: Alertdialog Pattern
  • Accessible Name and Description: Alert dialogs must include an accessible name via aria-labelledby and an accessible description via aria-describedby. 4.1.2 Name, Role, Value
  • Focus Management: When the alert dialog appears, focus must move to the first focusable element inside the dialog. Focus should be trapped until the dialog is dismissed. 2.4.3 Focus Order
  • Keyboard Operability: All interactive elements within the alert dialog must be accessible via keyboard. Users must be able to dismiss the dialog using the ESC key or an accessible dismiss button. 2.1.1 Keyboard
  • Interruptive Behavior: The alert dialog must have aria-modal="true" and prevent interaction with background content while it is active. 1.4.13 Content on Hover or Focus
  • Urgency Cues: Use semantic content and appropriate visual styling (e.g., warning icons, color contrast) to reinforce the urgency of the alert. Avoid relying on color alone to convey importance. 1.4.1 Use of Color

Role Attribute Element Usage
alertdialog div Identifies the element as an interruptive modal dialog requiring user response.
aria-labelledby="alertTitle" div Gives the alert dialog an accessible name by referring to the element that provides the dialog title.
aria-describedby="alertDescription" div Gives the alert dialog an accessible description by referring to the dialog content that describes the primary message or purpose of the dialog. (Optional)
aria-modal="true" div Informs assistive tech that background content is inactive.

  • Trigger and Dialog Styling: Display the alert dialog trigger button in its resting state and show the open dialog state with clear visual separation from the background. Include a visible heading and clear message that describes the alert's purpose.
  • Primary vs. Secondary Actions: Clearly distinguish between the primary action (e.g., Delete, Confirm) and secondary action (e.g., Cancel). Use distinct button styles and ensure tab order follows a logical reading flow.
  • Focus Behavior: The initial focus should go to the most prominent action (e.g., Delete button) when the dialog opens. Use annotations in Figma to show this focus behavior clearly for developer handoff.
  • Dismissal Options: Always include a Cancel or Close button. Show the ESC key functionality in the annotation or provide guidance for keyboard users to dismiss the dialog.
  • Content Clarity: Keep the text concise and easy to scan. Use clear headings, short paragraphs, and sufficient spacing to prevent cognitive overload—especially in high-stakes interactions like confirmations or warnings.
  • Non-reliance on Color Alone: Pair warning icons or color indicators with text labels to ensure users with color vision deficiencies can still understand the alert’s intent.

When auditing an Alert Dialog component within Figma, ensure the interaction pattern aligns with accessibility best practices for urgency, focus handling, and screen reader support.

Consider the following when reviewing alert dialogs during an accessibility audit:

  • Dialog Trigger and Title: Verify the triggering element is clearly defined in the design. The dialog should contain a prominent heading indicating the alert’s purpose (e.g., “Delete Item”).
  • Accessible Name and Description: Ensure the dialog includes a heading with a unique ID referenced via aria-labelledby, and any descriptive text is referenced with aria-describedby.
  • Focus on Launch: Confirm that the first actionable element (usually the primary action) receives focus when the dialog opens. Note in the audit if this behavior is not visually demonstrated in the design.
  • Traps Focus: Alert dialogs should trap keyboard focus until an action is taken. Use Figma annotations to clarify this behavior if it’s not visually represented.
  • Dismissal Mechanisms: There should be at least one secondary action (e.g., a Cancel button). Ensure the Escape key behavior is documented or visually represented.
  • Clarity of Messaging: Ensure critical information is conveyed clearly, not just by color. Use paired icons and text for warnings, errors, or destructive actions.
  • Open Dialog State: If only the trigger is shown in Figma, recommend that the designer include the full open dialog state for review and testing annotations.

Alert Dialog

Dialog Role, Name, and Description:

  • Must use role="alertdialog" on the dialog container.
  • Must include aria-modal="true" to indicate modal behavior.
  • Must have aria-labelledby that points to the heading element inside the dialog.
  • Should use aria-describedby to associate longer messages or warnings with the dialog.

Focus Management:

  • Focus must move to the first actionable element (usually a confirm/delete button) when opened.
  • Focus must be trapped inside the dialog while it is open.
  • Escape key must dismiss the dialog.
  • Focus must return to the triggering element after the dialog closes.

Note: Unlike standard modal dialogs, alert dialogs are designed to interrupt the user's task and typically require a decision before continuing. Annotations should clarify urgency and confirm that the dialog meets this behavioral expectation.

Alert dialogs must interrupt the user’s task with critical information or confirmation actions. To meet WCAG 2.1 AA and WAI-ARIA Authoring Practices, alert dialogs require the correct roles, ARIA labeling, and strict focus control.

Accessible Alert Dialog Code Example

This example shows a basic confirmation alert dialog that uses the required roles and ARIA relationships.


<button id="openAlertDialog" aria-haspopup="dialog" aria-controls="alertDialog">
  Delete Item
</button>

<div id="alertDialog"
    role="alertdialog"
    aria-modal="true"
    aria-labelledby="alertTitle"
    aria-describedby="alertDescription"
    class="modal"
    tabindex="-1">

  <h2 id="alertTitle">Delete Confirmation</h2>
  <p id="alertDescription">Are you sure you want to permanently delete this item? This action cannot be undone.</p>

  <button type="button" id="cancelBtn">Cancel</button>
  <button type="button" id="confirmBtn">Delete</button>

</div>

Tip: Use JavaScript to send focus to the first action button on open, trap focus while the dialog is visible, and return focus to the triggering element after it closes.

Key Developer Considerations

  • Role: Use role="alertdialog" for urgency. This notifies assistive tech that user input is required before continuing.
  • Modal Behavior: Always include aria-modal="true" to ensure the rest of the interface is marked as inert.
  • Accessible Name: Use aria-labelledby pointing to a heading inside the alert dialog (e.g., the alert title).
  • Accessible Description: Use aria-describedby to reference a paragraph or container that explains the action.
  • Triggering Element: The button that opens the dialog should use aria-haspopup="dialog" and clearly state the action.
  • Initial Focus: Focus should go to the most important action (e.g., the primary confirmation button) on open.
  • Focus Trap: Prevent users from tabbing out of the alert dialog until they take action or dismiss it.
  • Escape Key: Support dismissal using the Escape key, even though it may not always be visually represented.
  • Return Focus: After dismissal, return focus to the element that triggered the alert dialog unless it no longer exists.
  • Minimal Actions: Avoid overloading alert dialogs with multiple interactive elements. Stick to concise text and 1–2 clear buttons.
  • Test with AT: Validate with screen readers that the alert is announced when opened and all labels and roles are announced correctly.
