Listbox

A listbox is a widget that allows users to select one or more items from a list of options. It is commonly used in filters, settings panels, and selection interfaces where dropdowns or checkboxes aren’t appropriate. To be accessible, listboxes must support keyboard interaction, proper labeling, and clear indication of selection and focus.

Below is a list of WCAG success criteria requirements and accessibility considerations:

  • Proper Use of ARIA Roles and Properties: Implement role="listbox" for the container, role="option" for each list item, and aria-activedescendant to manage focus on the currently selected item. ARIA: Listbox Example
  • Keyboard Navigation: Ensure users can navigate the listbox options with keyboard input using arrow keys, select with Enter, and close the listbox (if applicable) with Escape. 2.1.1 Keyboard
  • Focus Management: Focus should be managed such that it remains within the listbox without disrupting workflow. The selected option should be clearly indicated visually and programmatically. 2.4.3 Focus Order
  • Labeling: Ensure the listbox and its options are correctly labeled using aria-labelledby to associate the listbox with its label, and aria-selected for the selected option. ARIA: Listbox Role
  • State Indication: Use aria-expanded (if applicable) to show whether the listbox is expanded or collapsed, providing users clear context about its current state. 1.3.1 Info and Relationships

W3C APG Listbox Patterns

Below is a list of different listbox patterns that demonstrate various implementations of the listbox design pattern:

  • Scrollable Listbox Example: Single-select listbox that scrolls to reveal more options, similar to HTML select with size attribute greater than one.
  • Example Listboxes with Rearrangeable Options: Examples of both single-select and multi-select listboxes with accompanying toolbars where options can be added, moved, and removed.
  • Listbox Example with Grouped Options: Single-select listbox with grouped options, similar to an HTML select with optgroup children.

Note: If a simple selection of options is required, using a native HTML <select> element is often the best choice for accessibility and ease of use. Here’s an example:

<label for="example-listbox">Choose an option:</label>
      <select id="example-listbox" multiple>
        <option>Option 1</option>
        <option>Option 2</option>
        <option>Option 3</option>
      </select>

  • Ensure Distinction: Make the listbox visually distinct with clear borders and ensure that each option within the listbox is easy to identify.
  • State Indicators: Design icons or visual indicators to clearly show when an option is selected, and ensure the selection state is prominent (e.g., through bold text, color change, or checkmarks).
  • Clear Labeling: Ensure the listbox is properly labeled in Figma, using appropriate labels for both the listbox and individual options to facilitate screen reader navigation.
  • Color Contrast: Maintain sufficient color contrast for the text and background in the listbox, as well as for the selected and non-selected states, to ensure readability for users with visual impairments.
  • Keyboard Focus: Design the focus state for the listbox so it's clearly visible when the user navigates with the keyboard (e.g., through a border or background change).

When auditing a listbox component in Figma, review its design elements for accessibility, focusing on visual compliance. Pay attention to how the listbox is structured and whether key accessibility features are visually present.

  • Color Contrast Compliance: Ensure that the text and background colors for the listbox and its options meet WCAG 2.1 AA contrast requirements (at least 4.5:1 for normal text and 3:1 for large text).
  • Visual Grouping: Ensure there is a clear visual distinction for grouped options within the listbox (if applicable), using headings, separators, or spacing.
  • Focus and Hover States: Verify that each option has clearly defined focus and hover states, using visual changes like border or background color to indicate interaction.
  • Selection States: Ensure that the selected option is visually distinct, using color, bold text, or other indicators to make it stand out from non-selected options.
  • Accessible Labeling: Confirm that the listbox has visible labels and instructions that clearly guide users on how to interact with it.

Listbox

  • Use appropriate HTML markup: Implement the role="listbox" for the container and role="option" for each item within the listbox.
  • Ensure keyboard accessibility: Users must be able to navigate the listbox using arrow keys, select options with Enter, and exit the listbox with Escape or Tab.
  • Manage focus state: Implement aria-activedescendant to manage the focus on the currently highlighted option. Ensure the focused option is visually distinct.
  • Maintain color contrast: Ensure all text and background colors meet WCAG 2.1 AA color contrast standards (4.5:1 for normal text, 3:1 for large text).
  • Define hover and focus states: Ensure hover and focus states are clearly indicated with visual changes such as background or border color.
  • Implement correct selection states: Use aria-selected="true" to indicate the selected option and ensure it is visually distinct from non-selected options.
  • Support groupings if applicable: If the listbox has groups, use role="group" and ensure they are visually distinguished with headers or separators.
  • Refer to ARIA Listbox Patterns: Follow the guidance from the APG Listbox Patterns for detailed accessibility practices.

  • Use ARIA Roles: The container element of the listbox should have role="listbox", and each item within the listbox should use role="option" to define individual selectable options.
  • Manage ARIA States: Use aria-selected="true" for the currently selected option within the listbox. This should update dynamically as the user selects different options.
  • Ensure Keyboard Navigation: Listboxes should be fully operable via keyboard. Use arrow keys to navigate between options, Enter to select an option, and Esc to exit the listbox without making a selection if applicable. Follow standard keyboard navigation patterns for listboxes.
  • Focus Management: Properly manage focus by ensuring that focus is visibly indicated as it moves through the listbox options. Use aria-activedescendant to identify which option is currently focused. Once an option is selected, ensure the correct focus behavior based on interaction pattern.
  • Link Label and Listbox: Ensure the listbox has an accessible label by using aria-labelledby to associate it with a visible label element. This ensures that assistive technologies can correctly announce the purpose of the listbox.
  • Maintain Color Contrast: Ensure that all text and background colors meet WCAG 2.1 AA contrast requirements (4.5:1 for normal text, 3:1 for large text) to ensure readability for users with visual impairments.
  • Define Hover and Focus States: Clearly indicate focus and hover states on listbox options using visible changes, such as background color or border, to show which option is currently focused or hovered over.
  • Group Options if Applicable: If the listbox includes groups of options, use role="group" to define each group and ensure there is a clear visual distinction between grouped options.
  • Screen Reader Behavior: Ensure screen readers correctly announce each listbox option as users navigate through them, and announce the currently selected option. Additionally, screen readers should notify users when the selection state changes.
  • Test Across Devices: Ensure that the listbox is tested across various devices, including mobile devices, to confirm the experience is consistent and accessible on touchscreens.
  • Follow ARIA Best Practices: For more information on accessible listboxes, consult the WAI-ARIA Authoring Practices for Listboxes.
