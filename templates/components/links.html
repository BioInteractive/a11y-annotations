Link

Button Vs. Link

A button <button> performs an action. (Open Disclosure, Play Video, Launch Dialog)

A link <a> navigates. (In-page anchor link, internal/external URL)

1.3.1 Info and Relationships: Information, structure, and relationships conveyed through presentation can be programmatically determined or are available in text.

A link is an interactive user interface element, typically created using the HTML <a>. An (anchor) link is a clickable element that allows users to navigate to another webpage, resource, or location by specifying a URL, enhancing the interconnectedness of web content.

2.4.4 Link Purpose: The purpose of each link can be determined from the link text alone or from the link text together with its programmatically determined link context, except where the purpose of the link would be ambiguous to users in general.

Links must be preceivable as links, must have an accessible name for link purpose understand where links lead without relying solely on additional context or non-descriptive text. Links must also be in focus order and must function equally for all users.

Ensuring Links are Distinguishable with Purpose

  • Distinguishable: Make the links distinguishable and styled differently than buttons.
  • Descriptive link text: Use clear, descriptive link text that describes the purpose and destination of the link.
  • Usability - Color Considerations: When styling the links, consider color use. If the link is in a block of text, ensure it is distinguishable with either a 3:1 contrast ratio with surrounding text or a non-color indicator for good usability.

Conducting Accesibility Audit and Annotations

  1. Use this A11y Annotation Kit.
  2. Identify all UI elements that navigate users.
  3. During phase one, ask questions if neccesary to understand navigation.
  4. During phase two in preparation for developer hand-off, add a sticky note annotation to define each link element as a link.
  5. Most links will require only minimal annotating for the developer hand-off phase. Please see interactives and the How to Annotate for more detailed instructions on how to number the elements and write annotations specific for more complex instances of link components.
  6. If the link does not have a visible text label it is recommended to use the link annotation tag with the additional label to ensure the link name will be communicated to the developer for programming.

Link

  • Link must be in a <a> element or the element used such as a <div> must have role="link" and tabindex="0" as an equivalent with keyboard event handlers ensuring equal functionality for mouse, keyboard and touch users.
  • Links must have an accessible name with good link purpose. Use text, img alt, aria-label, aria-labelledby or add sr-only text.
  • Links with ambiguous names such as "Learn More" and "Read More" will need additional context added with an aria-label starting with the link name or sr-only text or an aria-describedby association to add context to where the link will naviagte to.

Implementing Links for Accessibility

For developers, ensuring that links are accessible is crucial for navigating both in-page and external resources effectively:

  • Correct Element Usage: Use the <a> element for links with a valid href attribute to ensure they navigate as expected.
  • Descriptive Link Text: Provide descriptive text for each link that clearly indicates its purpose and destination, aiding in user understanding and accessibility tool compatibility.
  • Focusable Elements: Ensure links are focusable to facilitate keyboard navigation for users with mobility impairments or those who prefer keyboard controls.
  • Visual Distinction: Style links to be visually distinguishable from non-link text through color, underlining, or other stylistic differences without relying solely on color.
  • Accessible Name: Ensure links have an accessible name, which can be achieved through the link text itself or additional ARIA attributes or sr-only text when necessary.
  • Testing with Screen Readers: Test link accessibility using screen readers to ensure the link purpose is clear and navigation is intuitive for users relying on assistive technologies.

By adhering to these considerations, developers can create more accessible web content that supports a wider range of users, including those with disabilities, and complies with WCAG guidelines.
