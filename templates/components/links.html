Link

Button Vs. Link

A button <button> performs an action. (Open Disclosure, Play Video, Launch Dialog)

A link <a> navigates. (In-page anchor link, internal/external URL)

A link is an interactive element that allows users to navigate to another page, a different section of the same page, or an external resource. Commonly created using the <a> tag, links are essential to how users explore and interact with web content. To be accessible, links must be clearly identifiable, convey purpose, and be operable through a variety of input methods.

Below is a list of WCAG success criteria requirements and accessibility considerations:

  • Info and Relationships: Ensure the link's structure and surrounding relationships are programmatically determinable. 1.3.1 Info and Relationships
  • Link Purpose (In Context): The link’s purpose should be clear from the link text alone or with its programmatic context. 2.4.4 Link Purpose
  • Use of Color: When links are in blocks of text, use more than just color—like underlining or bold styling—to show they are interactive. 1.4.1 Use of Color
  • Identifiable as Interactive: Ensure links are visually styled to indicate interactivity (e.g., underlines, hover changes, or iconography).
  • Name, Role, Value: Every link should have a meaningful accessible name for screen reader users. 4.1.2 Name, Role, Value
  • Keyboard Navigation: Include all links in the tab order and ensure they are operable using keyboard controls. 2.1.1 Keyboard
  • Focus Visibility: Provide a clearly visible focus indicator for all links. 2.4.7 Focus Visible
  • Consistent Functionality: Links should behave consistently across contexts and input methods.
  • External Targets: If a link opens in a new tab or window, clearly indicate this in the visible link text or with an accessible label (e.g., visually hidden / sr-only text) so users are not surprised by the change in context.

Ensuring Links are Distinguishable with Purpose

  • Distinguishable: Make the links distinguishable and styled differently than buttons.
  • Descriptive link text: Use clear, descriptive link text that describes the purpose and destination of the link.
  • Usability - Color Considerations: When styling the links, consider color use. If the link is in a block of text, ensure it is distinguishable with either a 3:1 contrast ratio with surrounding text or a non-color indicator for good usability.

Conducting Accesibility Audit and Annotations

  1. Use this A11y Annotation Kit.
  2. Identify all UI elements that navigate users.
  3. During phase one, ask questions if neccesary to understand navigation.
  4. During phase two in preparation for developer hand-off, add a sticky note annotation to define each link element as a link.
  5. Most links will require only minimal annotating for the developer hand-off phase. Please see interactives and the How to Annotate for more detailed instructions on how to number the elements and write annotations specific for more complex instances of link components.
  6. If the link does not have a visible text label it is recommended to use the link annotation tag with the additional label to ensure the link name will be communicated to the developer for programming.

Link

  • Link must be in a <a> element or the element used such as a <div> must have role="link" and tabindex="0" as an equivalent with keyboard event handlers ensuring equal functionality for mouse, keyboard and touch users.
  • Links must have an accessible name with good link purpose. Use text, img alt, aria-label, aria-labelledby or add sr-only text.
  • Links with ambiguous names such as "Learn More" and "Read More" will need additional context added with an aria-label starting with the link name or sr-only text or an aria-describedby association to add context to where the link will navigate to.

Accessible links are essential for navigation and orientation. Follow the guidelines below to ensure your link implementations are programmatically determinable and perceivable by all users.

Accessible Link Code Example

This basic example shows a correctly structured link with descriptive text and an optional screen-reader-only span for additional context:


<a href="/resources/climate-data">
  View Climate Data 
</a>

Tip: Avoid vague link text like “Click here” or “Learn more.” Use visible, descriptive text that conveys destination or action.

For design purposes, if your link must visibly say “Learn more” or “Read more” due to space constraints, use an aria-label that begins with the visible text and includes additional context—for example, aria-label="Learn more about this dataset". This ensures screen reader users receive the full purpose of the link without changing its visual appearance.

Key Developer Considerations

  • Use Semantic Elements: Use <a> elements with a valid href for all navigable links. Do not use <div> or <span> with click handlers unless absolutely necessary.
  • Descriptive Link Text: Link text should clearly describe the link's purpose. Avoid generic phrases like “learn more.”
  • Visible Focus Styles: Ensure keyboard users can see which link has focus. Apply custom :focus styles that meet contrast requirements (at least 3:1).
  • Color Contrast: Links must have a 4.5:1 contrast ratio against the background and should be visually distinguishable from surrounding text using more than just color.
  • External Link Disclosure: If a link opens in a new tab or window, communicate this with screen-reader-only text.
  • Accessible Name Testing: Use Accessibility panel in Chrome Dev Tools to confirm the full accessible name of each link is accurate and informative.
  • Text Link Indicators: If a link is embedded within a block of text, it must include a non-color visual indicator such as an underline, bold styling, or icon to distinguish it from surrounding text. This ensures the link is perceivable for users with color vision deficiencies.
  • Avoid Redundant Links: Do not use multiple links in a row that point to the same destination with different text. Consolidate them for clarity as a best practice.
